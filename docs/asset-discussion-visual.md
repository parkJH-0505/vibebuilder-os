## 1. 현황 진단

### 1.1 현재 비주얼 에셋 인벤토리

| 에셋 유형 | 수량 | 분포 |
|----------|------|------|
| 이미지 (PNG/JPG/SVG 콘텐츠용) | **0개** | 없음 |
| Mermaid 다이어그램 | **6개** | ch02 (1), ch03 (3), ch04 (2) |
| 테이블 | ~20개 | 대부분의 챕터에 1~3개 |
| 커스텀 MDX 컴포넌트 | 5종 | 전 챕터 (Callout, SelfCheck, KeyTakeaway, ActionItem, NextPreview) |
| 외부 링크 (참고자료) | ~25개 | ch06~ch15에 집중 |

### 1.2 핵심 문제

**ch02~ch04에만 다이어그램이 있고, 나머지 12개 챕터(ch00~ch01, ch05~ch15)는 텍스트와 테이블만으로 구성되어 있다.** 이는 Part 2~5로 갈수록 독자의 시각적 단서가 급격히 줄어들어, 추상적 개념(MVP, 기술부채, 반복론 등)을 텍스트만으로 소화해야 하는 인지 부담을 가중시킨다.

특히 문제가 되는 지점:
- **개념적 메타포가 풍부하지만 시각화가 전무**: 식당 비유(ch01~02), 호텔 키카드 비유(ch04), 레고 블록 비유(ch05) 등 강력한 메타포가 텍스트로만 전달됨
- **프로세스/워크플로우 설명이 번호 목록으로만 존재**: PRD 작성 4단계(ch06), 빌드 루프(ch13), 배포 체크리스트(ch14) 등
- **비교/선택 가이드가 테이블에만 의존**: 도구 비교(ch09), 인증 서비스 비교(ch04), 외부 서비스 비교(ch05)

## 2. 에셋 유형 분류 및 제작 방법

### 2.1 에셋 유형 체계

이 프로젝트의 콘텐츠 성격(교육용, 비개발자 대상, 한국어, 개념 중심)에 맞는 에셋 유형을 6가지로 분류한다.

| ID | 에셋 유형 | 설명 | 주요 용도 |
|----|----------|------|----------|
| **D** | **다이어그램 (Mermaid)** | 코드로 생성하는 구조/흐름 다이어그램 | 데이터 흐름, 아키텍처, 의사결정 트리 |
| **I** | **일러스트 (AI 생성)** | Gemini/DALL-E로 생성하는 컨셉 일러스트 | 메타포 시각화, 챕터 대표 이미지, 감성적 앵커 |
| **S** | **SVG/CSS 인포그래픽** | 직접 제작하는 벡터 그래픽 | 비교 차트, 타임라인, 체크리스트, 매트릭스 |
| **C** | **캡처/스크린샷** | 실제 도구/서비스 화면 캡처 | 도구 UI, 설정 화면, 에러 메시지 예시 |
| **A** | **애니메이션/인터랙티브** | CSS 애니메이션 또는 React 컴포넌트 | 데이터 흐름 시뮬레이션, 단계별 진행 |
| **O** | **오픈소스 에셋** | undraw.co, heroicons 등 무료 리소스 | 범용 아이콘, 장식적 일러스트 |

### 2.2 제작 방법별 특성

| 제작 방법 | 장점 | 단점 | 적합한 경우 |
|----------|------|------|-----------|
| **Mermaid** | 코드로 관리, MDX에 직접 삽입, 수정 용이, 버전 관리 가능 | 시각적 커스터마이징 한계, 복잡한 레이아웃 불가 | 흐름도, 시퀀스, 의사결정 트리, 간단한 아키텍처 |
| **Gemini AI 생성** | 메타포/개념 시각화에 강함, 빠른 생성 | 일관성 유지 어려움, 텍스트 렌더링 부정확, 수정 불편 | 챕터 대표 이미지, 감성적 메타포, 비유 시각화 |
| **SVG/CSS 직접 제작** | 완전한 커스터마이징, 반응형, 테마 일관성 보장 | 제작 시간 많이 소요, 전문 기술 필요 | 비교 매트릭스, 체크리스트, 프로세스 다이어그램 |
| **스크린샷 캡처** | 가장 사실적, 독자가 실제 화면과 대조 가능 | 도구 업데이트 시 구식이 됨, 라이선스 확인 필요 | 도구 설정 가이드, 실제 에러 메시지 예시 |
| **오픈소스 에셋** | 무료, 즉시 사용, 라이선스 명확 | 프로젝트 고유의 컨셉에 정확히 맞지 않을 수 있음 | 장식적 아이콘, 범용 일러스트 보충 |

### 2.3 권장 제작 우선순위

1. **Mermaid 다이어그램** (최우선) — 코드 기반으로 즉시 삽입 가능, 유지보수 최적
2. **SVG/CSS 인포그래픽** — 핵심 비교/프로세스 시각화에 투자 가치 높음
3. **AI 생성 일러스트** — 메타포 시각화와 챕터 대표 이미지로 감성적 앵커 역할
4. **스크린샷** — 도구 관련 챕터(ch08, ch09)에서 선택적 사용
5. **오픈소스 에셋** — 빈 공간 보충용으로 가장 후순위

| 1-1 | **"무대 세트" 비포/애프터 일러스트** | I | "이거 진짜 내가 만든 거야?" 섹션 | 왼쪽: 영화 세트 정면 — 그럴듯한 건물 외관 / 오른쪽: 뒤에서 본 모습 — 나무 골조와 빈 공간만. 바이브코딩 결과물이 "정면에서만 그럴듯"하다는 핵심 인사이트의 직관적 전달 | 독자가 자신의 결과물을 객관적으로 바라보게 만드는 "아하" 모먼트 생성. 텍스트만으로는 "설마 내 것도?"라는 방어적 반응이 나오지만, 이미지는 즉각적 인식 전환 유도 | **P1** |

### Ch.2 — 프로덕트의 5가지 장기

**현재 상태**: Mermaid 다이어그램 1개 (User→FE→API→BE→DB 흐름). 테이블 1개 (도구 매핑).

| # | 에셋명 | 유형 | 위치 (섹션) | 설명 | 영향도 | 우선순위 |
|---|--------|------|-----------|------|--------|---------|
| 2-1 | **5가지 장기 아키텍처 일러스트** | I 또는 S | 챕터 도입부 | 인체 해부도 스타일: 5개 장기(프론트엔드=피부, API=신경계, 백엔드=뇌, DB=기억, 인프라=뼈대)를 인체에 매핑한 시각적 비유. 기존 Mermaid의 선형 흐름도(User→FE→API→BE→DB)와 상호 보완 | 챕터 제목이 "장기(臟器)"인데 실제로 장기처럼 시각화된 것이 없음. 이 일러스트가 있으면 "5가지 구성요소"가 유기적으로 연결된 하나의 생명체라는 핵심 메시지 전달 강화 | **P1** |
| 2-2 | **도구 매핑 매트릭스** | S | "나의 도구 지도" 섹션 | 5x5 격자: 행=도구(Lovable, Bolt, Cursor, Supabase, Vercel), 열=구성요소(FE, API, BE, DB, 인프라). 각 칸에 O/X 또는 강도 표시. 현재 텍스트 테이블의 시각적 업그레이드 | 현재 테이블이 이미 정보를 담고 있지만, 매트릭스 형태의 시각화는 "이 도구가 어디까지 커버하는지"를 한눈에 비교하게 함. 특히 "빈칸"이 시각적으로 드러나면서 "아, 이 도구만으론 부족하구나"를 체감 | P2 |
| 2-3 | **"Vertical Slice" 개념도** | D | "수직으로 관통하라" 섹션 | 5개 레이어를 수직으로 관통하는 화살표 1개(=vertical slice) vs 수평으로 1개 레이어만 넓게 펼친 화살표(=horizontal spread). "하나의 기능을 5개 레이어 모두에서 완성"하는 것의 시각적 대비 | "vertical slice"는 ch02의 핵심 전략이자 이후 ch07(MVP), ch13(빌드)에서 반복 참조됨. 시각화 없이는 이 개념이 추상적으로 남을 위험 | **P1** |

### 4.5 AI 생성 이미지 프롬프트 가이드라인

Gemini 또는 DALL-E로 메타포 일러스트를 생성할 때 일관성을 위한 가이드:

- **스타일**: Flat illustration, minimal detail, clean lines. 만화/카툰이 아닌 비즈니스 일러스트 톤
- **색상**: 위 색상 시스템 내에서. 과도한 그라데이션 지양
- **구도**: 중앙 집중, 여백 충분, 텍스트 오버레이 공간 확보
- **인물**: 포함 시 다양성 고려, 과도한 디테일 없이 실루엣이나 심플한 형태
- **텍스트**: AI 생성 이미지 내 텍스트는 부정확하므로, 이미지에 텍스트를 포함하지 않고 별도 라벨로 처리
- **배경**: 흰색 또는 매우 연한 배경. 복잡한 배경 지양

## 6. 크로스 챕터 시각 연결

### 6.1 반복 참조 에셋 (Recurring Visual Anchors)

아래 에셋들은 한 번 만들면 여러 챕터에서 반복 참조되므로, 투자 대비 효과가 가장 높다.

| 에셋 | 최초 등장 | 재참조 챕터 | 효과 |
|------|----------|-----------|------|
| 5가지 구성요소 (식당 매핑 또는 장기도) | ch01~02 | ch05, ch06, ch08, ch09, ch12 | "5가지 중 어디를 보고 있는지"의 지속적 위치 감각 |
| Vertical Slice 개념도 | ch02 | ch07, ch13 | "하나의 기능을 관통"하는 전략의 반복 강화 |
| 빌드 루프 | ch13 | ch14, ch15 | 빌드→배포→반복의 리듬 연속성 |
| 프롬프트 4요소 | ch10 | ch12, ch13 | AI와의 모든 대화에서 참조 |

### 6.2 Part 전환 시각 마커

각 Part(0~5)의 시작 지점에서 여정 로드맵(0-2)을 축소 버전으로 재활용하여, "현재 위치"를 업데이트하면 독자의 전체 여정 내 위치 감각을 유지할 수 있다.

- Part 0 (ch00): "점화" 하이라이트
- Part 1 (ch01~05): "지도 그리기" 하이라이트
- Part 2 (ch06~08): "직접 만들기" 하이라이트
- Part 3 (ch09~11): "실전 도구와 대화법" 하이라이트
- Part 4 (ch12~13): "마스터리" 하이라이트
- Part 5 (ch14~15): "검증과 성장" 하이라이트

### 핵심 메시지

1. **ch02~04에 집중된 Mermaid 6개만으로는 16개 챕터를 시각적으로 지탱할 수 없다.** 특히 ch00~01(도입부)과 ch05~15(본론)의 시각적 공백이 심각하다.

2. **가장 시급한 것은 ch01의 핵심 프레임워크 시각화**(무대 세트, 빙산, 식당 매핑). 이 3개 에셋이 이후 15개 챕터의 시각적 참조 기반이 된다.

3. **Mermaid 다이어그램을 Phase 1으로 즉시 실행하면, 외부 도구 없이 7개 에셋을 추가하여 다이어그램 커버리지를 2배 이상으로 확대할 수 있다.**

4. **SVG/CSS 인포그래픽이 전체의 48%를 차지하며, 이것이 가장 큰 투자 항목이다.** React 컴포넌트로 만들면 재사용성과 반응형을 모두 확보할 수 있다.

5. **크로스 챕터 반복 참조 에셋**(5가지 구성요소, Vertical Slice, 빌드 루프, 프롬프트 4요소)은 한 번의 투자로 여러 챕터의 시각적 품질을 동시에 향상시키므로 ROI가 가장 높다.
