## "localhost:3000"을 카톡으로 보냈습니다

Ch.7에서 기능을 절반으로 줄이고, MVP를 만들었습니다. 이제 내보낼 차례입니다.

어느 일요일 밤, 드디어 됩니다. 화면이 뜨고, 버튼이 작동하고, 데이터가 저장됩니다. 흥분한 마음에 친구한테 카톡을 보냅니다.

> "야, 나 앱 만들었어! 여기 들어가봐 → localhost:3000"

5분 뒤 답장이 옵니다.

> "아무것도 안 뜨는데?"

당연합니다. **localhost는 "내 컴퓨터"라는 뜻**이거든요. 그 주소는 당신의 노트북 안에서만 작동합니다. 친구의 폰에서 localhost:3000을 치면, 친구의 폰 안에 있는 서버를 찾는 건데 — 거기엔 아무것도 없습니다.

Ch.1에서 잠깐 나왔던 이야기 기억나시나요? "내 컴퓨터에서만 되는 것과, 다른 사람도 접속할 수 있는 것은 다르다"고. 바로 그 차이입니다.

localhost에서 돌아가는 건 **연습**입니다. 진짜 서비스가 되려면, 이걸 **인터넷 어딘가에 올려서 누구나 접속할 수 있게** 만들어야 합니다.

이 과정을 **배포(Deploy)**라고 합니다. 그리고 이 챕터는, 그 배포의 커튼 뒤를 들여다보는 시간입니다.

다들 처음엔 이 낭패를 경험합니다. 이제 그걸 알고 시작하는 사람이 됐으니, 절반은 먹고 들어가는 겁니다.

## 배포는 "마법의 버튼"이 아닙니다

Vercel이라는 플랫폼에 GitHub 계정을 연결하고, "Deploy" 버튼을 누르면... 3분 뒤에 진짜 URL이 생깁니다. `https://my-app.vercel.app` 같은 주소가 나오고, 전 세계 어디서든 접속 가능합니다.

"와, 이렇게 쉬워?" 맞습니다, 표면적으로는 쉽습니다. 근데 이 3분 동안 뒤에서 일어나는 일을 이해해야, **문제가 생겼을 때 당황하지 않습니다.** 그리고 문제는 반드시 생깁니다.

배포 버튼을 누르는 건 쉽습니다. 하지만 그 뒤에는 **환경변수, 도메인, DNS, SSL, CI/CD**라는 톱니바퀴들이 맞물려 돌아가고 있습니다.

이 톱니바퀴들을 중요한 순서대로 알아보겠습니다.

## 첫 번째이자 가장 중요한 톱니바퀴: 환경변수

배포 후 가장 많이 듣는 말이 있습니다.

> "로컬에서는 되는데, 배포하니까 안 돼요."

바이브코딩 커뮤니티에서 **"통곡의 벽"**이라고 불리는 현상입니다. 이 벽에 부딪히는 원인의 대부분은 하나입니다.

<Callout type="warning">
배포 실패의 80% 이상은 단 하나의 이유입니다 — 배포 플랫폼에 환경변수를 등록하지 않은 것. .env 파일은 보안을 위해 GitHub에 올라가지 않기 때문에, Vercel은 API Key를 모릅니다. 코드가 완벽해도 환경변수가 없으면 아무것도 작동하지 않습니다.
</Callout>

**배포 플랫폼에 환경변수를 등록하지 않은 것.**

배포 실패의 **80% 이상이 이 하나의 이유**입니다. 나머지 이유를 다 합쳐도 이것보다 적습니다.

Ch.5에서 배운 걸 떠올려봅시다. API Key 같은 민감한 정보는 코드에 직접 쓰지 않고 `.env` 파일에 넣는다고 했죠? 이 `.env` 파일은 **보안을 위해 GitHub에 올라가지 않습니다.** `.gitignore`에 등록되어 있으니까요.

그 결과 무슨 일이 벌어지냐면, 내 컴퓨터에는 `.env` 파일이 있어서 API Key를 읽을 수 있고 작동하지만, GitHub에는 `.env` 파일이 없고, Vercel은 GitHub에서 코드를 가져오므로 API Key를 모릅니다. 그래서 작동하지 않습니다.

해결법은 간단합니다. **Vercel(또는 Netlify) 대시보드에서 환경변수를 직접 등록**하면 됩니다. `.env` 파일에 있던 Key-Value를 배포 플랫폼의 설정 화면에 하나씩 옮겨 적는 겁니다.

식당 비유로 말하면, `.env` 파일은 **내 금고에 보관한 거래처 코드**입니다. 식당을 새 건물(Vercel)로 이전했는데, 거래처 코드를 금고에서 꺼내서 새 건물 금고에 넣어주지 않으면 식재료 주문이 안 되는 것과 같습니다.

실제로 이런 일이 벌어집니다. 한 바이브코더가 첫 배포를 했는데 아무것도 작동하지 않았습니다. 카카오 로그인 버튼을 누르면 에러, 이미지 업로드를 하면 에러, 데이터를 저장하면 에러. 세 시간 동안 코드를 뜯어봤는데 코드는 멀쩡했습니다. 알고 보니 Vercel에 환경변수를 한 개도 등록하지 않은 것이었습니다. 등록하고 나니 전부 작동했습니다.

여기서 다들 한 번씩 막힙니다. 이제 막힐 이유가 없습니다.

### 배포 전 환경변수 체크리스트

배포 버튼을 누르기 **전에** 이것만 확인하세요:

- [ ] `.env` 파일에 있는 변수 목록을 전부 확인했는가?
- [ ] 배포 플랫폼의 환경변수 설정에 **모든 변수를 등록**했는가?
- [ ] 로컬에서 `localhost:5000` 같은 주소를 하드코딩한 곳이 없는가?
- [ ] Supabase URL, API Key 등 외부 서비스 정보가 빠짐없이 등록되었는가?

이 체크리스트 하나면 "통곡의 벽"을 미리 예방할 수 있습니다.

<SelfCheck question="배포 직후 카카오 로그인이 작동하지 않습니다. 코드는 로컬에서 정상이었습니다. 가장 먼저 확인해야 할 것은 무엇인가요?" hint="로컬과 배포 환경의 차이가 무엇인지 생각해보세요.">
Vercel 대시보드의 환경변수 설정을 가장 먼저 확인합니다. 카카오 로그인에 필요한 API Key, Client ID, Redirect URL 등이 배포 플랫폼에 등록되어 있는지 확인하세요. .env 파일의 모든 변수가 배포 환경에도 동일하게 등록되어 있어야 합니다.
</SelfCheck>

## 두 번째 톱니바퀴: 도메인과 DNS — 인터넷의 연락처 앱

배포를 하면 Vercel이 `my-app.vercel.app` 같은 주소를 자동으로 만들어줍니다. 근데 진짜 서비스를 하려면 **자기만의 주소** — `dongnae-market.com` 같은 — 가 필요합니다.

이 주소를 **도메인(Domain)**이라고 합니다.

근데 컴퓨터는 `dongnae-market.com`이라는 이름을 이해하지 못합니다. 컴퓨터가 이해하는 건 `76.76.21.21` 같은 숫자(IP 주소)뿐입니다. 그러면 `dongnae-market.com`을 쳤을 때 `76.76.21.21`로 연결해주는 무언가가 필요합니다. 이걸 **DNS(Domain Name System)**라고 합니다.

가장 쉬운 비유는 **스마트폰의 연락처 앱**입니다.

연락처에서 "김철수"를 터치하면 폰이 자동으로 010-1234-5678로 전화를 겁니다. 마찬가지로, 브라우저에서 "dongnae-market.com"을 입력하면 DNS가 자동으로 76.76.21.21로 연결합니다.

### 도메인 연결은 3단계입니다

**1단계: 도메인 구매**
GoDaddy, Namecheap, 가비아 같은 도메인 등록 업체에서 원하는 이름을 삽니다. `.com`은 연 1~2만 원 정도입니다.

**2단계: DNS 설정 변경**
도메인 등록 업체의 관리 페이지에서, "이 도메인은 Vercel의 서버를 가리킨다"라고 설정합니다. Vercel이 어떤 값을 넣을지 화면에서 알려줍니다.

**3단계: 전파 대기**
설정 후 바로 안 될 수 있습니다. DNS 설정이 전 세계 서버에 퍼지는 데 **수 분에서 최대 48시간**이 걸립니다. 이걸 모르면 "설정했는데 왜 안 되지?" 하고 당황하게 됩니다.

아는 사람은 그냥 기다립니다. 이게 다입니다.

## 세 번째 톱니바퀴: SSL — 봉인된 편지

도메인을 연결하면 주소가 생깁니다. 근데 그 주소가 `http://`로 시작하나요, `https://`로 시작하나요?

이 차이가 **SSL**(Secure Sockets Layer) — 인터넷 데이터를 암호화하는 보안 프로토콜 — 인증서의 유무입니다.

비유를 하나 드립니다.

- **HTTP** = 투명한 엽서. 누구나 내용을 볼 수 있습니다.
- **HTTPS** = 봉인된 편지. 보내는 사람과 받는 사람만 내용을 볼 수 있습니다.

사용자가 로그인 비밀번호를 입력하거나, 결제 정보를 넣을 때, HTTP로 보내면 **그 정보가 중간에서 그대로 노출될 수 있습니다.** HTTPS는 이 데이터를 암호화해서, 중간에 누가 가로채도 읽을 수 없게 만듭니다.

다행히 Vercel이나 Netlify는 **SSL 인증서를 무료로 자동 발급**해줍니다. 배포하면 자동으로 `https://`가 붙습니다.

<Callout type="info">
Ch.4의 인증, Ch.5의 API Key 보안은 모두 HTTPS라는 봉인된 편지 위에서 작동합니다. SSL이 없으면 나머지 보안이 다 허공에 뜨는 것과 같습니다. Vercel과 Netlify는 이 인증서를 무료로 자동 발급하므로, 배포하는 순간 자동으로 https://가 붙습니다.
</Callout>

Ch.4에서 배운 인증, Ch.5에서 배운 API Key 보안을 떠올려보세요. 이 모든 보안이 **HTTPS라는 봉인된 편지** 위에서 작동하는 겁니다. SSL이 없으면 나머지 보안이 다 허공에 뜹니다.

## 네 번째 톱니바퀴: CI/CD — 자동 배포 파이프라인

환경변수를 등록하고, 첫 배포를 성공했다고 칩시다.

그 다음날 버그를 발견합니다. 코드를 수정합니다. 이제 또 배포해야 합니다. 매번 Vercel에 가서 "Deploy" 버튼을 누르나요?

아닙니다. Vercel에 GitHub를 연결했으면, **코드를 GitHub에 올리는 것만으로 자동 배포**됩니다. 코드를 수정하고 GitHub에 올리면, 몇 분 뒤에 사이트가 자동으로 업데이트됩니다.

이 자동화를 **CI/CD**(Continuous Integration / Continuous Deployment) — 지속적 통합/지속적 배포라고 합니다.

피자 주방의 **자동화 조립 라인**이라고 생각하면 됩니다.

과거(수동 배포)에는 셰프가 도우 만들기부터 포장, 배달까지 직접 했습니다. 매번. CI/CD에서는 셰프가 레시피(코드)를 컨베이어 벨트에 올리면(GitHub에 push하면), 로봇이 검사→빌드→테스트→배포까지 자동으로 처리합니다.

이게 왜 중요하냐면요. 바이브코딩을 하다 보면 하루에도 여러 번 코드를 수정합니다. 매번 수동으로 배포하면 시간도 걸리고 실수도 생깁니다. CI/CD는 "코드 올리면 끝"이라는 편리함을 줍니다.

CI/CD가 있으면 **"먼저 배포하고 나중에 고친다"**는 전략이 가능해집니다. Ch.7에서 말한 "기능 하나 → 테스트 → 배포" 사이클을 실천하려면 배포가 쉬워야 합니다. CI/CD는 그 쉬운 배포를 보장하는 인프라입니다.

## 배포 플랫폼 선택 — Vercel vs Netlify

"그래서 어디에 배포하면 되나요?"

바이브코딩으로 만든 프로젝트를 배포하는 가장 흔한 플랫폼 두 가지가 있습니다.

| 항목 | Vercel | Netlify |
|------|--------|---------|
| **강점** | Next.js/React 프로젝트에 최적화. "Auto-detect"로 설정 거의 불필요 | 정적 사이트, JAMstack에 강함. 내장 폼, 인증 기능 |
| **무료 빌드 시간** | 월 6,000분 | 월 300분 |
| **무료 대역폭** | 100GB | 100GB |
| **상업적 이용** | 무료 플랜에서 금지 | 무료 플랜에서 허용 |
| **커스텀 도메인** | 무료 | 무료 |
| **SSL 인증서** | 자동 무료 | 자동 무료 |

바이브코딩으로 만든 프로젝트가 대부분 Next.js나 React 기반이라면(Cursor, Claude Code, Lovable 등에서 만든 코드), **Vercel이 가장 마찰이 적습니다.** GitHub 연결하고 "Deploy" 누르면 끝입니다.

다만, **상업적 서비스**로 발전시킬 계획이 있고 아직 유료 플랜을 쓸 단계가 아니라면, Netlify의 무료 플랜이 상업적 이용을 허용하므로 선택지가 될 수 있습니다.

어떤 걸 선택하든, Ch.5에서 배운 원칙을 떠올리세요. 도구는 바뀔 수 있습니다. 중요한 건 **"코드를 올리면 빌드-테스트-배포가 자동으로 돌아간다"는 CI/CD의 원리**를 이해하는 것입니다.

## "Deploy Early" — 완성 후가 아니라 Day 1에 배포하세요

많은 분이 "다 만들고 나서 배포해야지"라고 생각합니다. 이건 위험합니다.

코드가 100줄일 때 배포하면, 에러가 나면 어디가 문제인지 바로 찾을 수 있습니다. 코드가 10,000줄일 때 처음 배포하면, 에러가 어디서 나는지 **추적이 불가능**합니다.

바이브코딩 전문가들이 공통적으로 하는 조언입니다:

<Callout type="tip">
"프로젝트 뼈대만 잡히면, 그 상태로 먼저 배포부터 하세요." 빈 화면이라도 좋습니다. 코드가 100줄일 때 배포해야 에러 추적이 가능합니다. 10,000줄이 되고 나서 처음 배포하면, 에러가 어디서 나는지 추적이 불가능해집니다.
</Callout>

빈 화면이라도 좋습니다. Vercel에 연결하고, URL을 받고, "아, 이 구조에서 배포가 된다"는 걸 확인하세요. 그 다음부터는 기능을 하나 추가할 때마다 자동으로 배포되니까, "기능 하나 → 테스트 → 배포" 사이클이 자연스럽게 돌아갑니다.

## 마법 뒤의 톱니바퀴를 이해하면 달라지는 것

지금까지 배운 4가지 톱니바퀴를 정리합니다.

| 순서 | 톱니바퀴 | 역할 | 식당 비유 |
|------|---------|------|----------|
| 1 | **환경변수** | API Key 등 민감 정보를 배포 환경에 전달 | 새 건물 금고에 거래처 코드 넣기 |
| 2 | **도메인 + DNS** | 이름(dongnae-market.com)을 주소(IP)로 변환 | 식당 간판과 전화번호부 등록 |
| 3 | **SSL** | 데이터를 암호화하여 안전하게 전달 | 배달 음식을 밀봉 포장 |
| 4 | **CI/CD** | 코드 변경 시 자동으로 빌드-배포 | 레시피 바꾸면 자동으로 새 메뉴 반영 |

순서가 중요합니다. 배포 첫날에 신경 써야 하는 건 **환경변수**입니다. 도메인과 DNS는 그 다음, SSL은 플랫폼이 자동으로 해결합니다.

이 톱니바퀴들을 몰라도 배포 버튼은 누를 수 있습니다. 하지만 **알고 누르는 사람**은 다릅니다.

- DNS 전파에 48시간 걸려도 당황하지 않습니다
- "로컬에서 되는데 왜 안 돼?"에 10초 만에 "환경변수 확인해보세요"라고 답합니다
- `http://`와 `https://`의 차이가 왜 중요한지 설명할 수 있습니다
- CI/CD 덕분에 "고치고 → 올리면 끝"이라는 워크플로우를 자연스럽게 씁니다

<SelfCheck question="도메인을 구입하고 DNS를 설정했는데 30분이 지나도 사이트가 열리지 않습니다. 무슨 일인가요? 어떻게 해야 하나요?" hint="DNS 전파 특성을 떠올려보세요.">
정상입니다. DNS 설정이 전 세계 서버에 전파되는 데 수 분에서 최대 48시간이 걸립니다. 기다리면 됩니다. 확인하고 싶다면 'DNS propagation checker' 도구로 전파 상태를 실시간으로 볼 수 있습니다. 이 시간 동안 코드 작업을 계속하면 됩니다.
</SelfCheck>

Ch.1에서 시작한 여행이 여기까지 왔습니다. 프로덕트가 뭘로 이루어져 있는지 알고(Part 1), 아이디어를 기획서로 바꾸고(Ch.6), 핵심만 골라내고(Ch.7), 세상에 내보내는 것(Ch.8)까지.

URL이 생겼습니다. `https://`가 붙었습니다. 전 세계 어디서든 접속할 수 있습니다. 이건 더 이상 연습이 아닙니다. **진짜 서비스가 됐습니다.**

- 배포 실패의 대부분은 환경변수 누락이다
- 도메인, DNS, SSL은 배포의 4가지 톱니바퀴다
- Deploy Early — 완성 전에 배포하고, 배포하면서 완성하라

</KeyTakeaway>
