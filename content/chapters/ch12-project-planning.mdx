"아이디어는 있는데... 뭘 먼저 하지?"

이 챕터는 그 멈춤을 3단계로 풉니다. 코드 한 줄 쓰기 전에 해야 할 모든 것을 여기서 끝냅니다.

## 1. 첫 프로젝트의 함정: 왜 대부분 두 번째에서 성공할까

첫 프로젝트에서 가장 흔한 실패 원인, 뭘까요?

"실력 부족"이 아닙니다. **"너무 많이 만들려는 것"**입니다.

어떤 사람은 첫 프로젝트로 창업자와 투자자를 연결하는 매치메이킹 플랫폼을 시도했습니다. 복잡한 알고리즘이 필요한 서비스. 결과요? 디버깅 지옥에 빠져서 포기했습니다. 또 어떤 사람은 인기 앱의 클론을 만들려다가 "기능이 왜 이렇게 많아?"라는 현실 앞에서 무너졌습니다.

야심이 가장 큰 적입니다.

AI가 만든 결과물은 "돌아가는 것처럼 보이는" 상태를 빠르게 만들어냅니다. 한 비개발자는 45분 만에 앱이 나왔다고 좋아했습니다. 하지만 전문가에게 보여주자마자 수많은 허점이 드러났습니다. 이건 "진짜 프로덕트"가 아니라 "데모"입니다. Ch.1에서 배운 빈 껍데기와 같은 이야기입니다.

이렇게 조언한 사람이 있습니다.

<Callout type="tip">
"야심 찬 아이디어가 있다면, 먼저 한 세션 안에 만들어보세요. 대부분의 기능이 가짜 데이터로 채워져 있고 제대로 작동하지 않는 현실을 깨닫게 될 겁니다. 이것이 기대치를 현실적으로 만드는 첫걸음입니다."
</Callout>

솔직히 말하면, 대부분의 사람들이 **첫 번째 프로젝트에서 "이런 거였구나"를 배우고, 두 번째 프로젝트에서 비로소 성공합니다.** 이 챕터의 목표는 그 시행착오를 한 번 줄이는 것입니다.

## 2. 좋은 첫 프로젝트의 5가지 조건

이 챕터의 핵심 기준을 먼저 말하겠습니다.

**첫 프로젝트는 완성도가 아니라 완주율을 최적화한다.**

멋진 걸 만드는 게 아닙니다. 끝까지 가보는 게 목표입니다. 그리고 성공적으로 완주한 사람들에게는 일관된 패턴이 있습니다. 거창한 사업 아이템이 아니라, **나를 위한 작고 재미있는 도구**에서 시작한 사람들이 끝까지 갑니다.

트레이더 조 과자를 좋아하는 어떤 사람은, 분위기와 식이 제한을 입력하면 스낵 박스를 추천해주는 간단한 웹 앱을 만들었습니다. 코드 한 줄 직접 수정하지 않고 완성했습니다. 또 누군가는 수백 개의 스크린샷 파일명을 자동으로 바꿔주는 스크립트를 만들었습니다. 둘 다 "나만 쓰는 작은 도구"였고, 둘 다 완성되었습니다.

이 사례들을 관통하는 **좋은 첫 프로젝트의 5가지 조건**입니다.

| # | 조건 | 이유 |
|---|------|------|
| 1 | **나만 쓴다** | 다른 사용자를 고려할 필요 없음. 에러가 나도 나만 불편 |
| 2 | **서버 배포 불필요** | "남들이 접속하는 서버" 없이, 내 컴퓨터에서만 돌아감 |
| 3 | **로그인 불필요** | 인증 시스템은 첫 프로젝트에서 가장 시간 잡아먹는 함정 |
| 4 | **목표가 프로토타입** | "완벽한 서비스"가 아니라 "돌아가는 시연용"이면 성공 |
| 5 | **브라우저에서 실행** | 서버 없이 클라이언트 측에서만 돌아가면 가장 안전 |

<Callout type="warning">
로그인/회원가입 기능은 첫 프로젝트에서 가장 자주 프로젝트를 죽이는 기능입니다. 인증 시스템은 겉으로 간단해 보여도 보안, 세션 관리, 에러 처리가 얽혀 있어 삽질이 길어집니다. "로그인이 꼭 필요한가?"를 먼저 물어보세요. 대부분의 첫 프로젝트는 없어도 됩니다.
</Callout>

### 포스트잇 테스트

아이디어를 빠르게 검증하는 방법이 있습니다.

**포스트잇 한 장에 그 아이디어를 적어보세요.** "이 앱은 ___을/를 입력하면 ___을/를 보여준다." 한 문장으로.

들어가지 않으면? 들어갈 때까지 줄이세요.

- "좋아하는 음식을 입력하면 레시피를 추천한다" — 포스트잇에 들어갑니다.
- "사용자가 가입하고 레시피를 공유하고 댓글을 달고 인기 레시피를 추천하고 결제하면 프리미엄 레시피를 볼 수 있는 플랫폼" — 이건 사업계획서입니다.

<SelfCheck question="지금 머릿속에 있는 아이디어를 '이 앱은 ___을 입력하면 ___을 보여준다' 한 문장으로 적을 수 있나요?" hint="한 문장으로 들어가지 않으면, 아직 MVP가 아닙니다. 기능 하나를 더 빼보세요.">
포스트잇 한 문장으로 정리된 아이디어와, 위 5가지 조건에 체크 3개 이상인 아이디어가 좋은 첫 프로젝트 후보입니다.
</SelfCheck>

> **✅ 1막 완료 — 뭘 만들지 골랐습니다.** 아이디어를 포스트잇 한 문장으로 줄이고, 5가지 조건으로 검증했습니다. 이제 어떤 도구로 만들지 정할 차례입니다.

## 3. 기술 스택 선택: 단계에 맞는 도구를 쓴다

Ch.9에서 바이브코딩 도구 생태계를 배웠습니다. 이제 실제로 고를 차례입니다.

핵심 원칙은 하나입니다. **프로젝트의 "단계"에 따라 도구가 달라진다.**

### 검증 단계 → 프로덕션 단계

| 단계 | 목적 | 도구 (2026-02 기준) | 핵심 |
|------|------|---------------------|------|
| **검증** | 아이디어가 말이 되는지 확인 | Lovable, Bolt.new | 속도. 완벽할 필요 없음 |
| **프로덕션** | 실제 사용자에게 서비스 | Cursor + Supabase + Vercel | 제어권. 보안/확장성 확보 |

검증 단계에서는 Lovable이나 Bolt.new로 시작하세요. 프롬프트 몇 줄이면 화면이 나옵니다. 이 결과물은 **"계속 발전시킬 가치가 있나?"를 판단하기 위한 것**입니다.

아이디어가 검증되어 진짜 서비스를 만들 때는 도구를 바꿔야 합니다. 프로토타입 도구로 만든 결과물에는 보안, 성능, 확장성이 빠져 있습니다. Ch.11에서 배운 기술 부채가 바로 여기서 시작됩니다.

핵심은 도구 이름이 아닙니다. **지금 내가 검증 단계인지, 프로덕션 단계인지 아는 것**이 도구 선택의 전부입니다.

### 첫 프로젝트 스택 선택 — 간단한 기준

| 만들고 싶은 것 | 검증 단계 | 프로덕션 단계 |
|----------------|-----------|---------------|
| 간단한 웹 앱 / 도구 | Lovable 또는 Bolt | Cursor + Supabase |
| 랜딩페이지 / 포트폴리오 | v0 by Vercel | Vercel 배포 |
| SaaS (구독 서비스) | Lovable + Supabase | Cursor + Supabase + Vercel |
| AI 기능이 핵심인 앱 | Bolt 또는 Replit | Cursor + 외부 AI API |

백엔드가 필요하다면 **Supabase로 시작**하세요. 대부분의 바이브코딩 도구가 Supabase와의 자동 연동을 지원하고, 무료 플랜이 넉넉합니다.

<Callout type="info">
**결제 연동에 대해:** 결제는 비개발자에게 가장 어려운 부분입니다. 다만 상황이 바뀌고 있습니다. 토스페이먼츠가 MCP 서버를 도입해서 자연어로 결제 연동이 가능해졌습니다 (2026-02 기준). 하지만 첫 프로젝트 범위에는 넣지 마세요. 두 번째, 세 번째에서 도전하는 게 현실적입니다.
</Callout>

> **✅ 2막 완료 — 어떤 도구로 만들지 정했습니다.** 지금 단계(검증 vs 프로덕션)를 판단하고, 그에 맞는 도구를 골랐습니다. 이제 마지막, AI에게 넘길 기획서를 쓸 차례입니다.

## 4. 기획서(PRD) 작성: AI에게 넘기는 설계도

도구를 골랐으면, 이제 가장 중요한 단계입니다. **기획서를 쓰는 것.**

Ch.6에서 PRD를 배웠습니다. Ch.11에서 "건축가처럼 생각하라"고 했습니다. 이제 그것을 실제 문서로 만듭니다.

### 왜 기획서가 먼저인가

"그냥 AI한테 바로 말하면 안 돼요?"

되긴 됩니다. 근데 대부분 실패합니다.

기획 없이 "소셜 미디어 앱 만들어줘"라고 하면 AI는 자기 맘대로 해석합니다. 그럴듯해 보이지만 내가 원하던 것과 다릅니다. 수정하면 다른 부분이 깨집니다. Ch.11에서 본 두더지 게임이 시작됩니다.

기획서가 있으면? "이 문서 따라서 1단계부터 만들어줘." AI가 길을 잃지 않습니다. 여러분이 길을 잃지 않습니다.

<SelfCheck question="기획서의 '만들지 않을 것' 목록이 '만들 것' 목록보다 짧다면 어떤 문제가 생길까요?" hint="첫 프로젝트에서 범위가 커지는 건 대부분 '만들지 않을 것'을 명확히 안 했기 때문입니다.">
'만들지 않을 것' 목록이 명확할수록 프로젝트는 완주될 가능성이 높아집니다. 이 목록이 야심과 싸우는 가장 효과적인 무기입니다.
</SelfCheck>

### 바이브코딩용 PRD: Ch.6을 실전으로

Ch.6에서 PRD의 6가지 구성요소를 배웠습니다 — 목표(Why), 유저 스토리(Who & What), 기능 범위(Scope), 기술 스택, AI 지침, 구현 계획. 구조는 같습니다. **달라지는 건 두 가지입니다.**

**차이 1: "만들지 않을 것" 칸이 생존의 열쇠가 된다**

Ch.6에서는 개념으로 배웠습니다. 실전에서는 이 칸이 프로젝트의 생사를 결정합니다. "있으면 좋겠는데"라고 생각한 것의 절반을 여기에 넣으세요. 위 5가지 조건(나만 쓴다, 서버 배포 불필요, 로그인 불필요...)에 맞지 않는 기능은 전부 "만들지 않는다" 칸입니다.

**차이 2: 구현 계획이 "수직적 슬라이스"로 바뀐다**

전체를 한 번에 만들라고 하지 않습니다. **기능 단위로 세로로 얇게 쪼깁니다.** 각 단계가 "화면에서 데이터까지 완결된 하나의 기능"이 됩니다.

> **1단계:** 재료 입력 화면 만들기 (UI만, 백엔드 연결 없음)
> **2단계:** 입력한 재료를 AI API에 보내서 레시피 3개 받아오기
> **3단계:** 레시피 결과를 카드 형태로 보여주기
> **4단계:** 카드를 클릭하면 상세 레시피 보기

1단계만 끝내도 "화면이 뜬다", 2단계까지 하면 "AI가 답한다" — 진행이 눈에 보이니까 동기부여가 유지됩니다. 이게 완주율을 높이는 핵심입니다.

## 5. 실전 연습: PRD 직접 써보기

지금까지 배운 것을 하나로 합쳐봅시다. 아래는 실제로 AI에게 넘길 수 있는 PRD의 전체 예시입니다.

```markdown
# PRD: 냉장고 레시피 추천기

## 1. 목표
냉장고에 있는 재료로 만들 수 있는 요리를 AI가 추천해주는 개인용 웹 앱.
매일 "오늘 뭐 먹지" 고민을 줄이는 것이 목표.

## 2. 타겟 사용자
- 나 혼자 사용 (배포 불필요)
- 요리 초보, 매일 재료 조합에 고민하는 사람

## 3. 유저 스토리
- 사용자는 냉장고 재료를 텍스트로 입력한다
- 입력 후 "추천받기" 버튼을 누르면 AI가 레시피 3개를 추천한다
- 레시피 카드를 클릭하면 상세 조리법을 볼 수 있다

## 4. MVP 범위
### 만든다
- 재료 입력 폼
- AI 레시피 추천 (3개)
- 레시피 상세 보기

### 만들지 않는다
- 로그인/회원가입
- 레시피 저장
- 다른 사용자 공유
- 장보기 목록

## 5. 기술 스택
- 프레임워크: Next.js (App Router)
- 스타일: Tailwind CSS
- AI: OpenAI API (gpt-4o-mini)
- 배포: 로컬 개발 서버 (배포 안 함)
- 언어: TypeScript, 한국어 UI

## 6. AI 지침
- 컴포넌트당 200줄 이하
- 주석은 한국어
- console.log 금지
- 에러 메시지는 사용자 친화적 한국어

## 7. 구현 단계 (Vertical Slice)
### Phase 1: 입력 UI
- 재료를 쉼표로 구분하여 입력하는 폼
- "추천받기" 버튼
- 입력값 유효성 검사 (빈 값 방지)

### Phase 2: AI 연동
- 입력된 재료를 OpenAI API에 전달
- 프롬프트: "다음 재료로 만들 수 있는 한국 가정식 레시피 3개를 JSON으로"
- 로딩 상태 표시

### Phase 3: 결과 표시
- 레시피 3개를 카드 UI로 표시
- 카드에 요리 이름, 난이도, 예상 시간 포함

### Phase 4: 상세 보기
- 카드 클릭 시 모달로 상세 조리법 표시
- 재료 목록, 단계별 조리법 포함
```

이 문서를 AI에게 넘기면서 이렇게 말합니다:

"이 PRD를 읽고 Phase 1부터 시작해줘. Phase 1이 완료되면 멈추고 결과를 보여줘."

한꺼번에 다 만들라고 하지 않습니다. 한 단계씩, 확인하면서 진행합니다.

**오늘 당장 1가지만 한다면:** 만들고 싶은 아이디어 하나를 잡고, 위 PRD 예시처럼 6가지 구성요소로 적어보세요. 완벽하지 않아도 됩니다. 30분이면 충분합니다. 그게 코드 한 줄보다 더 중요한 첫걸음입니다.

> **✅ 3막 완료 — AI에게 넘길 설계도가 완성되었습니다.** 뭘 만들지 골랐고, 어떤 도구로 만들지 정했고, 기획서까지 작성했습니다. 코드를 쓸 준비가 된 것입니다.
