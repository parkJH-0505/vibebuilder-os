## 사건: "한 줄도 직접 쓰지 않았다"는 자랑이 재앙이 된 날

2026년 2월, 'Moltbook'이라는 서비스가 뉴스에 떴습니다. AI 에이전트들의 소셜 네트워크. 꽤 근사한 아이디어였죠. 창업자는 자랑스럽게 밝혔습니다. "단 한 줄의 코드도 직접 작성하지 않았다. AI로 전부 만들었다."

보안 연구자들이 들여다봤습니다. 결과 — 150만 개의 API 인증 토큰과 3만 5천 개의 이메일 주소가 누구나 열람할 수 있는 상태였습니다. 외부 서비스(Supabase)를 연결해놓고, 그 연결을 안전하게 관리하는 방법을 몰랐던 겁니다.

이게 남의 이야기처럼 느껴질 수도 있습니다. 하지만 GitHub에 실수로 API 키를 올린 개발자에게 **4시간 만에** 8,900만 원짜리 AWS 요금 청구서가 날아오기도 합니다. GitHub에는 24시간 자동으로 돌아가는 봇들이 있어서, 코드가 올라가는 순간 몇 초~몇 분 안에 유출된 키를 찾아냅니다.

이 챕터는 두 가지를 가르쳐줍니다. **어떤 외부 서비스들이 있고 어떻게 고르는지**, 그리고 **그것들을 안전하게 연결하는 법**. Moltbook처럼 되지 않으려면, 둘 다 알아야 합니다.

---

<Callout type="warning">
Moltbook 사건은 예외가 아닙니다. GitHub에는 24시간 자동으로 돌아가는 봇들이 있어서, API 키가 코드에 올라가는 순간 몇 초~몇 분 안에 찾아냅니다. 외부 서비스를 연결할 줄 아는 것과, 안전하게 연결할 줄 아는 것은 다른 실력입니다.
</Callout>

## 발상 전환 — "배달의민족은 위성 사진을 직접 찍지 않는다"

챕터 4에서 인증 서비스를 외부에서 가져다 붙이는 걸 배웠습니다. 사실 그건 빙산의 일각이었습니다.

배달의민족을 뜯어볼게요. 우리가 매일 쓰는 이 앱이 실제로 뭘로 만들어져 있는지:

- **결제:** 토스페이먼츠(또는 유사 PG사) — 카카오페이, 네이버페이, 카드 결제 처리
- **지도:** 카카오맵/네이버맵 API — 가게 위치 표시, 배달 경로 계산
- **알림:** Firebase Cloud Messaging — "주문 접수됨", "배달 출발" 푸시 알림
- **파일 저장:** 클라우드 스토리지 — 가게 사진, 메뉴 이미지

배민이 이 기능들을 전부 직접 만들었을까요? 위성 사진을 직접 촬영하는 음식 배달 앱은 없습니다. 이미 잘 만들어진 서비스들을 조합한 겁니다.

노션도 마찬가지입니다. "AI로 요약해줘" 기능은 노션이 자체 AI를 만든 게 아니라, OpenAI 모델을 API로 연동한 것입니다. 첨부 파일은 AWS S3에 저장하고, 초대 메일은 이메일 전송 서비스가 보냅니다.

**프로덕트를 만드는 건 코드를 처음부터 짜는 게 아니라, 이미 있는 서비스들을 골라서 조합하는 것입니다.**

식당이 세탁을 세탁 업체에 맡기고, 배달은 배달대행에 맡기고, 결제는 카드 단말기 회사에 맡기는 것처럼요. 빨래를 직접 하거나 배달 오토바이를 사는 건 말이 안 됩니다. 이미 잘하는 전문 업체가 있으니까요.

## 6가지 카테고리 — "이런 걸 하고 싶다면"

"이런 기능이 필요한데..."라는 생각이 들 때, 열에 아홉은 이미 누군가가 서비스로 만들어놓은 겁니다.

### "돈을 받고 싶다" — 결제

결제를 직접 구현한다는 건 카드 정보 암호화, 보안 인증, 환불 처리, 세금 계산, 사기 방지까지 전부 직접 만든다는 뜻입니다. 챕터 4에서 "인증은 직접 만들지 마라"고 했죠? 결제는 그보다 더합니다.

| 서비스 | 핵심 특징 | 수수료 | 언제 쓰면 좋은가 |
|---|---|---|---|
| **Stripe** | 전 세계 표준 결제 API. 125개 이상 결제 수단 지원. AI 도구들이 가장 잘 이해하는 결제 서비스 | 2.9%+$0.30 | 글로벌 서비스 |
| **토스페이먼츠** | 카카오페이, 네이버페이, 토스 등 한국 간편결제 전부 지원 | ~3.63% | 한국 사용자 대상 서비스 |
| **Paddle** | 전 세계 부가세·판매세를 알아서 처리해줌. 세금 처리를 맡기고 싶다면 유일한 선택 | 5%+$0.50 | 글로벌 SaaS, 세금 자동 처리 필요 시 |

### "자동 이메일을 보내고 싶다" — 이메일 발송

회원가입 인증, 비밀번호 재설정, 결제 영수증 — 서비스가 자동으로 보내는 이메일은 Gmail에서 직접 보내는 게 아닙니다.

| 서비스 | 핵심 특징 | 무료 범위 |
|---|---|---|
| **Resend** | 현재 가장 추천. API가 간결하고 현대적. 바이브코딩 도구와 호환성 최상 | 월 3,000건 영구 무료 |
| **Postmark** | 이메일 전달률 최상. 금융·법률 서비스처럼 스팸함에 빠지면 안 되는 경우에 최적 | 월 100건 영구 무료 |
| **AWS SES** | 압도적으로 저렴한 대량 발송. 단, 초기 설정이 매우 복잡하여 바이브코딩만으로 해결 어려움 | 월 3,000건 (12개월) |

바이브코딩으로 이메일 기능을 처음 붙인다면 **Resend**이 가장 현실적입니다. 월 3,000건이면 MVP 단계에서 넘칠 정도입니다.

### "AI를 내 서비스에 넣고 싶다" — AI 기능

챗봇, 텍스트 요약, 이미지 생성 — AI 기능을 내 서비스에 탑재하려면 AI API를 연동합니다.

| 서비스 | 핵심 강점 | 초기 무료 크레딧 |
|---|---|---|
| **OpenAI API** | 가장 범용적. 생태계가 가장 넓음. 바이브코딩 도구들의 사실상 표준 | 신규 $5 크레딧 |
| **Anthropic API (Claude)** | 코딩·논리 추론·긴 문서 분석에서 최고 성능 | 신규 $5 크레딧 |
| **Google Gemini API** | 텍스트+이미지+영상+음성 동시 처리. Google 생태계 연동 | 영구 무료 티어 + $300 크레딧 |

범용 AI 챗봇·콘텐츠 생성은 생태계가 넓은 OpenAI로 시작하고, 코드 생성·복잡한 문서 분석은 Anthropic(Claude), 이미지·영상 등 다양한 데이터를 동시에 처리해야 한다면 Google Gemini를 선택하면 됩니다.

### 파일 저장 · 지도 · 알림

- **파일 저장:** Supabase를 이미 쓴다면 Supabase Storage가 가장 자연스럽습니다. 대규모라면 AWS S3 또는 전송 비용이 없는 Cloudflare R2를 고려하세요.
- **지도:** 글로벌 서비스는 Google Maps API, 한국 서비스는 카카오맵 API (국내 도로명 주소·장소 데이터가 훨씬 정밀합니다).
- **알림:** 모바일/웹 푸시 알림은 Firebase Cloud Messaging(FCM)이 사실상 표준이고 무료입니다. SMS·카카오 알림톡이 필요하다면 Twilio.

---

<Callout type="tip">
새로운 기능을 추가하고 싶을 때 "이 기능은 이미 만들어진 서비스가 있나?"를 먼저 물어보세요. 결제, 인증, 이메일, 지도, AI, 파일 저장, 알림 — 이것들은 전부 외부 서비스를 쓰는 게 맞습니다. 내 서비스만의 고유 비즈니스 로직만 직접 만드세요.
</Callout>

## 디지털 열쇠와 금고 — API Key를 안전하게 다루는 법

이 외부 서비스들을 내 프로덕트에 연결하려면, 각 서비스에서 발급받는 것이 있습니다. **API Key**(API 키)입니다.

식당 비유로 말하면 — 외주 업체와 거래하려면 **거래처 코드**가 있어야 합니다. "여기 ○○식당인데 수거 부탁합니다" 할 때 등록된 코드를 대야 하죠. 이 코드가 있어야 주문이 들어가고, 이 코드로 사용량이 기록되고, 이 코드로 요금이 청구됩니다.

API Key가 정확히 이겁니다. 외부 서비스에 접근할 때 "나는 허가받은 사용자입니다"를 증명하는 고유 코드입니다.

### 외부 서비스 호출의 흐름

챕터 3에서 배운 데이터의 여행에 외부 서비스가 끼어들면 경로가 한 단계 더 늘어납니다. 사용자가 "결제하기"를 클릭하면 → 프론트엔드가 우리 서버로 요청을 보내고 → 우리 서버(백엔드)가 API Key와 함께 Stripe에 요청을 보내고 → Stripe이 처리 결과를 우리 서버로 돌려주면 → 사용자 화면에 "결제 완료!"가 표시됩니다.

**핵심: API Key는 반드시 백엔드에서만 사용해야 합니다.** 프론트엔드(사용자 브라우저)에 API Key가 노출되면, 누구든 브라우저 개발자 도구를 열어 키를 볼 수 있습니다. 그러면 아무나 내 키로 외부 서비스를 호출할 수 있게 됩니다.

### 이걸 잘못 다루면 진짜 큰일 납니다

| 서비스 | 유출 경로 | 실제 피해 |
|---|---|---|
| **AWS** | GitHub에 키 실수로 푸시 | 4시간 만에 **$89,000(약 1.2억 원)** — 공격자가 암호화폐 채굴에 악용 |
| **OpenAI** | 공개 저장소에 키 노출 | 수십만 달러 요금 — 유출된 키로 고비용 AI 모델 무단 호출 |
| **Stripe** | 프론트엔드 코드에 Secret Key 노출 | 사기 거래 발생, 고객 결제정보 유출 |
| **Moltbook** | 클라이언트 코드에 키 노출 + RLS 미설정 | 150만 개 API 키·개인정보 통째 유출 |

식당 거래처 코드가 적힌 종이를 식당 입구에 붙여놓으면 어떻게 되겠습니까? 아무나 그 코드로 주문을 넣을 수 있겠죠. 세탁물이 엉뚱한 곳으로 가고, 요금은 전부 우리한테 청구됩니다.

### 두 가지 파일만 알면 됩니다 — `.env`와 `.gitignore`

**`.env` 파일** — API Key를 저장하는 금고. 코드에 키를 직접 적는 대신, 이 별도 파일에 키를 보관합니다. 코드에서는 "금고에서 꺼내와"라는 방식으로 키를 사용합니다. 예를 들어 코드에 Stripe 키를 직접 적는 대신, `.env`에 보관하고 코드에서는 그 이름으로만 참조합니다.

**`.gitignore` 파일** — "이 파일은 GitHub에 절대 올리지 마"라고 적어두는 빨간 딱지. `.env`가 실수로 공개 저장소에 올라가는 것을 막아줍니다.

`.env`는 비밀 금고이고, `.gitignore`는 "이사할 때 이 금고는 트럭에 싣지 마세요"라고 적어둔 딱지입니다. 이 딱지를 빼먹으면 금고째로 인터넷에 공개됩니다.

가장 흔한 실수 — `.gitignore`에 `.env`를 추가하는 걸 깜빡하는 것. 깜빡한 순간, GitHub의 자동화 봇들이 유출된 키를 몇 분 안에 찾아냅니다.

### API Key 보안 — 반드시 지킬 네 가지

**1. 권한 최소화** — 키에 필요한 최소한의 권한만 부여합니다. 읽기만 필요하면 쓰기 권한은 끄세요. Stripe에서 테스트할 때는 반드시 "테스트 키"를 쓰고, "라이브 키"는 실제 서비스에만 사용합니다.

**2. 정기적 교체(30~90일)** — 주기적으로 키를 재발급하세요. 유출됐더라도 피해를 제한할 수 있습니다.

**3. 사용하지 않는 키 삭제** — 테스트용으로 만든 키를 방치하면 그것도 공격 경로가 됩니다.

**4. 사용량 한도 설정** ← 이게 핵심입니다. 대부분의 서비스에서 월간 사용량 상한을 설정할 수 있습니다. 키가 유출돼도 피해 금액에 상한이 생깁니다. **이것 하나만 설정해도 요금 폭탄의 99%를 막을 수 있습니다.**

<Callout type="info">
"Merchant of Record(MoR)"가 뭔가요? Paddle이 "판매 대행자" 역할을 하면 Paddle이 판매자가 되어, 유럽 VAT나 미국 주별 판매세 계산·납부를 전부 대신 처리합니다. 수수료가 비싼 이유가 이것 — 세금 회계사 비용이 포함된 겁니다.
</Callout>

## 요금 — "언제부터 돈이 나가는 건지"

외부 서비스를 쓰면 공짜가 아닌 것 아니냐는 걱정은 당연합니다. 하지만 대부분이 초기에는 충분한 무료 사용량을 제공합니다. 문제는 그 구조를 모르고 쓰다가 예상치 못한 요금이 나오는 겁니다.

**무료 티어** — 일정 사용량까지 무료. Resend은 월 3,000건, Google Gemini API는 영구 무료 티어가 있습니다. MVP를 만들고 검증하는 단계에서는 이것만으로 충분한 경우가 대부분입니다.

**종량제** — 쓴 만큼만 냅니다. OpenAI API가 대표적. 처리한 텍스트 양 단위로 과금됩니다.

진짜 조심해야 할 함정들:

**AI API 요금 급증** — AI 챗봇을 만들면, 대화 기록이 길어질수록 매 요청에 포함되는 텍스트 양이 기하급수적으로 늘어납니다. 처음엔 요청 하나에 0.01달러도 안 되다가, 대화가 50턴쯤 쌓이면 0.5달러가 될 수도 있습니다.

**비싼 모델 자동 사용** — 바이브코딩 중 AI가 더 비싼 모델을 쓰도록 코드를 짜는 경우가 있습니다. GPT-4o mini를 쓰려고 했는데 코드에는 GPT-4o가 박혀 있으면 요금 차이가 **10배 이상**. 코드에 어떤 모델 이름이 적혀 있는지 반드시 확인하세요.

<SelfCheck question="내가 만들고 싶은 서비스에서 AI API를 사용한다면, 어떤 요금 구조로 과금되고 폭탄을 막으려면 어떤 설정이 필요한가요?" hint="종량제 vs 무료 티어 구조를 떠올려보고, 사용량 한도 설정과 모델 이름 확인이라는 두 가지 포인트를 기억하세요.">
AI API는 처리한 텍스트 양(토큰) 기준으로 종량제 과금됩니다. 폭탄을 막는 두 가지 핵심은: 첫째, 서비스 대시보드에서 월간 사용량 한도(Spending Limit)를 반드시 설정할 것. 둘째, 바이브코딩으로 생성된 코드에 어떤 모델 이름이 박혀 있는지 확인할 것 — GPT-4o와 GPT-4o mini의 요금 차이는 10배 이상입니다.
</SelfCheck>

## 연동 전 체크리스트

외부 서비스를 붙이기 전에, 한 가지만 기억하세요. AI가 코드를 잘 짜도 **보안 설정은 내가 직접 확인해야 합니다.** 이 리스트를 한 번 훑어보는 데 2분이면 되고, 그 2분이 API Key 유출 사고를 막아줍니다.

- [ ] API Key를 `.env` 파일에 저장했다 (코드에 직접 적지 않았다)
- [ ] `.gitignore`에 `.env`가 포함되어 있다
- [ ] API Key가 프론트엔드가 아닌 백엔드에서만 사용된다
- [ ] 서비스의 요금 정책을 확인했다
- [ ] 월간 사용량 한도를 설정했다
- [ ] 테스트 모드에서 먼저 확인했다 (특히 결제!)
- [ ] 배포 환경(Vercel, Railway 등)에도 환경 변수를 따로 설정했다 (`.env`는 로컬에만 있으니까)

이 체크리스트를 AI에게 직접 보여주면서 "이것들 다 확인해줘"라고 지시하는 것도 좋은 방법입니다. 하지만 최종 확인은 반드시 내가 직접 합니다.

<SelfCheck question="외부 서비스를 새로 연동하려고 합니다. 코드를 짜기 전에 반드시 확인해야 할 보안 관련 3가지는 무엇인가요?" hint="API Key가 어디에 저장되는지, 어디에서 호출되는지, 그리고 실수로 GitHub에 올라가지 않도록 막는 장치를 생각해보세요.">
세 가지 핵심: 첫째, API Key는 코드에 직접 쓰지 않고 .env 파일에 저장한다. 둘째, .gitignore에 .env를 등록해서 GitHub에 절대 올라가지 않게 한다. 셋째, API Key는 프론트엔드가 아닌 백엔드에서만 호출한다 — 프론트엔드에 노출되면 브라우저 개발자 도구로 누구나 볼 수 있습니다.
</SelfCheck>

- API Key는 집 열쇠와 같다 — 절대 공개하지 마라
- 무료 티어를 활용하되, 요금 한도를 반드시 설정하라
