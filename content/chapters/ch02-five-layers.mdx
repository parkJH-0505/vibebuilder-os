## 질문이 시작됩니다

챕터 1에서 약속한 것이 있습니다. "당신이 가진 도구들이 5가지 구성요소 중 어디를 채우는지 지도 위에 꽂아드리겠다"고요.

지금부터 그 약속을 지킵니다.

할일 앱을 하나 만들어봤다고 해봅시다. AI 빌더에게 "할일 앱 만들어줘"라고 했더니 예쁜 화면이 나왔습니다. 할일을 입력하고, 체크하고, 삭제할 수 있는 화면. 여기까지는 좋습니다.

그런데 머릿속에 질문들이 하나씩 올라오기 시작합니다.

- "할일을 저장하려면 어디에?" — 데이터베이스가 필요합니다.
- "내 할일과 남의 할일을 구분하려면?" — 로그인과 백엔드 로직이 필요합니다.
- "입력한 할일이 서버에 전달되려면?" — API가 필요합니다.
- "친구한테 링크 보내서 같이 쓰려면?" — 배포가 필요합니다.

이 질문들에 답할 수 없으면, AI한테 "안 되는데 고쳐줘"만 반복하게 됩니다. AI도 뭐가 문제인지 정확히 모르니까 이것저것 바꿔보다가 오히려 더 꼬입니다. 처음엔 다 이렇습니다.

반대로, 각 구성요소가 뭘 하는지 알면 이야기가 완전히 달라집니다. "API 요청은 나가는데 데이터베이스에 저장이 안 된다" — 이 한 문장이면 AI가 정확히 어디를 고쳐야 하는지 압니다. "안 되는데"와 "API는 되는데 DB가 안 된다"의 차이가, 삽질 3일과 해결 30분의 차이를 만듭니다.

5가지 구성요소를 하나씩 열어봅시다.

## 프론트엔드 — "이건 이미 해봤죠"

**프론트엔드**(Frontend)는 사용자가 직접 보고 만지는 화면입니다. 버튼, 텍스트, 이미지, 색상, 레이아웃 — 화면에 보이는 모든 것이 프론트엔드입니다.

바이브코딩을 해봤으면, 사실 이건 이미 경험한 부분입니다. AI 빌더한테 "이런 앱 만들어줘"라고 하면 나오는 그 예쁜 화면 — 그게 프론트엔드입니다.

이 화면은 세 가지 재료로 만들어집니다.

- **HTML** — 화면의 "뼈대". "여기에 제목이 있고, 그 아래 버튼이 있다"는 구조.
- **CSS** — 화면의 "옷". "이 버튼은 파란색이고, 둥글고, 그림자가 있다"는 스타일.
- **JavaScript** — 화면의 "근육". "이 버튼을 누르면 이런 일이 일어난다"는 동작.

이 셋을 직접 쓸 일은 없습니다. "구조 / 스타일 / 동작" — 이 세 역할이 있다는 감만 잡으면 됩니다.

### 그런데 왜 React, Next.js 같은 이름이 자꾸 들릴까요?

화면이 복잡해지면 비슷한 부품을 매번 처음부터 만들어야 합니다. 로그인 화면, 대시보드, 설정 페이지... 솔직히 말이 안 되죠. **React**는 이 문제를 해결합니다 — 화면 부품을 블록처럼 만들어놓고, 필요할 때 가져다 조립하는 방식입니다.

**Next.js**는 React에 추가 기능을 얹은 것입니다. 화면을 서버에서 미리 조립해서 보내주는 기능이 핵심인데, 이게 있으면 구글 검색에도 잘 잡히고, API 기능도 같이 넣을 수 있어서 별도 백엔드 서버가 필요 없는 경우가 많습니다.

"왜 하필 React야?" 싶었다면 — **AI가 가장 많이 학습한 방식이라서** 그렇습니다. 수백만 개의 React 프로젝트를 학습했으니, 자연히 그 방식으로 코드를 가장 잘 만들어냅니다.

**이걸 왜 알아야 하냐면요.** AI 빌더가 뭔가를 만들어줬을 때, "예쁜 화면이 나왔다"에서 끝나면 안 되기 때문입니다. "이건 5가지 중 첫 번째(프론트엔드)가 만들어진 거구나. 나머지 네 개는?" — 이 생각이 자동으로 떠올라야 합니다.

## API — 주문서 없이 소리지르는 식당

이번엔 상상해봅시다. 식당 홀에서 손님이 주방을 향해 소리칩니다. "파스타요!" 그런데 주방은 못 알아듣습니다. 몇 인분인지, 어떤 파스타인지, 매운 건 빼달라는 건지 — 정해진 양식이 없으니까 혼란입니다.

이게 **API**가 없는 앱입니다.

프론트엔드에서 "이 할일을 저장해줘"라고 요청하고 싶습니다. 근데 아무렇게나 보내면 안 됩니다. 정해진 규격이 필요합니다. 이 규격이 **REST API**입니다.

### REST API의 4가지 구성요소

REST API 요청에는 4가지가 담깁니다. 식당의 주문서에 빈칸이 4개 있는 것과 같습니다.

**1. URL (엔드포인트)** — "어디로 보내는가"

`/todos`, `/users/123`, `/orders` 같은 주소입니다. 기능마다 주소가 다릅니다.

**2. HTTP 메서드** — "무엇을 하는가"

이론적으로는 이렇게 정리됩니다. 딱 네 가지밖에 없어서 외우기도 쉽습니다.

| 메서드 | 의미 | 할일 앱 예시 |
|---|---|---|
| GET | 가져와 | 내 할일 목록을 보여줘 |
| POST | 새로 만들어 | 할일을 추가해 |
| PUT | 수정해 | 이 할일 내용을 바꿔 |
| DELETE | 삭제해 | 이 할일을 지워 |

실제로 당신이 쓸 때는 이것만 기억하면 됩니다. "할일 목록 보여줘"는 `GET /todos`, "새 할일 추가해"는 `POST /todos`. 주소와 행동의 조합으로 모든 요청이 이루어집니다. 에러가 났을 때 AI에게 "GET이냐 POST냐"를 구분해서 말할 수 있는 것만으로 이미 충분합니다.

**3. Headers (헤더)** — "부가 정보"

"이 요청을 보내는 사람이 누구인지" 같은 메타 정보가 담깁니다. 가장 중요한 것은 **인증 토큰** — "이 사람은 로그인한 회원이다"를 증명하는 디지털 신분증입니다. 챕터 4에서 자세히 다룹니다.

**4. Body (본문)** — "실제 데이터"

POST나 PUT으로 뭔가를 만들거나 수정할 때, 실제 내용을 담는 곳입니다. "빨래하기"라는 할일을 추가하면, 할일 제목과 완료 여부 같은 정보가 **JSON**이라는 형식으로 담겨 보내집니다. JSON은 "키:값" 쌍으로 데이터를 표현하는 방식인데, 프로그래밍 세계의 공용어 같은 겁니다.

### 직접 만들 필요는 없지만, 알아야 하는 이유

Supabase를 쓰면 데이터베이스 테이블을 만드는 순간 REST API가 자동으로 생성됩니다. 직접 API를 만들 일은 거의 없습니다.

**중요한 건, 문제가 터졌을 때입니다.** "데이터가 저장이 안 돼요"라는 상황에서 — "그냥 안 돼요"라고 하면 AI도 막막합니다. 근데 "GET 요청은 잘 되는데 POST에서 에러가 나는 것 같아"라고 하면? AI가 정확히 어디를 봐야 하는지 바로 압니다. 직접 겪어보면 이 차이가 얼마나 큰지 알게 됩니다.

## 백엔드 — 주방에서 판단하는 것들

**백엔드**(Backend)는 주문서를 받아서 실제로 판단하고 처리하는 곳입니다. "이 사람이 정말 로그인한 회원인지 확인", "할일 목록에서 이 사람 것만 골라내기", "완료된 할일 개수 세기" 같은 **규칙과 로직**이 여기서 돌아갑니다.

### 서버란 뭘까요?

별거 아닙니다. **24시간 켜져 있는 컴퓨터**입니다. 여러분 노트북도 켜놓고 인터넷에 연결하면 서버가 될 수 있습니다. 다만 24시간 안정적으로 돌려야 하니까 전문 업체의 컴퓨터를 빌려 쓰는 겁니다.

사용자가 웹 브라우저에 주소를 치면, 그 서버가 웹사이트 파일을 보내줍니다. 클라이언트(당신의 브라우저)가 요청하면, 서버가 응답합니다. 식당에서 손님이 주문하면 주방이 요리를 내보내는 것과 같습니다.

### 서버리스 — "내가 관리할 필요 없는 서버"

"그럼 나도 서버를 관리해야 하는 거야?" 싶을 수 있습니다. 아닙니다. 서버를 직접 관리하는 건 보통 일이 아닙니다. 업데이트도 해야 하고, 고장나면 새벽에도 달려가야 하고. 그래서 등장한 것이 **서버리스**(Serverless)입니다.

이름이 "서버가 없다"는 뜻 같지만, 실제로는 **"내가 서버를 관리하지 않아도 된다"**는 뜻입니다.

- **직접 서버 운영** = 집에 발전기를 사서 전기를 직접 만드는 것
- **서버리스** = 전력회사의 전기를 쓴 만큼만 요금 내는 것

발전소(서버)는 어딘가에 있지만, 내가 관리할 필요는 없습니다. 코드 조각을 올려놓으면, **요청이 올 때만 실행되고 끝나면 꺼집니다**. 쓴 만큼만 비용이 발생합니다.

**바이브코더한테 서버리스는 거의 기본값입니다.** Supabase나 Vercel을 쓰는 순간 이미 서버리스를 사용하고 있는 겁니다. 직접 서버를 사거나 관리할 일은 없습니다. "서버가 필요하다"는 말에 겁먹을 이유가 없습니다.

## 데이터베이스 — 모든 것을 기억하는 곳

**데이터베이스**(Database, DB)는 정보를 저장하고 꺼내는 곳입니다. 할일 앱이라면 "할일 내용, 완료 여부, 누가 만들었는지, 언제 만들었는지"가 여기에 기록됩니다.

데이터베이스에는 크게 두 종류가 있습니다.

### SQL(관계형 데이터베이스) — 엑셀 시트

엑셀 시트를 떠올리면 됩니다. 정해진 열(이름, 이메일, 가입일)이 있고, 한 줄이 한 명의 데이터입니다.

핵심은 **테이블 간의 관계**를 만들 수 있다는 것입니다. 고객 테이블과 주문 테이블이 따로 있지만, "주문 #45는 고객 #12가 했다"라는 연결이 가능합니다. 그리고 **"데이터가 반만 처리되는 사고"를 방지**합니다 — 은행 이체에서 A 계좌에서 빠졌는데 B에 안 들어가는 일이 없도록 보장하는 것이죠. 그래서 금융, 커머스, 고객 관리 같은 서비스에는 SQL이 기본입니다. Supabase가 쓰는 **PostgreSQL**이 대표적입니다.

### NoSQL(비관계형 데이터베이스) — 자유로운 스크랩북

형식이 자유로운 스크랩북에 가깝습니다. 어떤 페이지에는 사진이, 어떤 페이지에는 메모가. 형식이 자유로워서 빠르게 바뀌는 데이터에 유리하고, 속도가 중요한 서비스에 강합니다. Firebase의 **Firestore**가 대표적입니다.

### 어떤 걸 골라야 할까?

"그래서 나는 뭘 써야 하는데?" 싶을 겁니다. 판별법은 간단합니다:

| 이런 서비스라면 | 추천 | 이유 |
|---|---|---|
| 고객 관리, 주문 시스템, 커머스 | **SQL** (Supabase) | 데이터 간 관계가 중요하고 정확성이 생명 |
| 채팅, 실시간 피드, 빠른 프로토타입 | **NoSQL** (Firebase) | 구조가 자주 바뀌고 속도가 중요 |

큰 서비스들은 둘 다 씁니다. **인스타그램**은 사용자 계정 정보는 PostgreSQL(SQL)로, 수백만 개의 좋아요/댓글은 Cassandra(NoSQL)로 관리합니다.

바이브코딩에서 Supabase(SQL)가 더 자주 추천되는 데는 이유가 있습니다. 데이터 구조가 명확하니 AI가 다루기 좋거든요. "users 테이블에서 email이 이것인 사람을 찾아줘"처럼 정형화된 요청을 AI가 잘 처리합니다.

## 인프라/배포 — 문을 여는 것

**인프라**(Infrastructure)와 **배포**(Deployment)는 만든 것을 인터넷에 올려서 다른 사람이 접근할 수 있게 만드는 기반입니다.

먼저 두 상태를 구분해야 합니다:

- **로컬(localhost)** = 내 컴퓨터에서만 돌아가는 상태. 나만 볼 수 있습니다.
- **프로덕션(Production)** = 인터넷에 올라간 상태. 누구나 접속할 수 있습니다.

"localhost:3000"이라는 주소를 본 적 있을 겁니다. 처음엔 "이 주소를 다른 사람한테 보내면 되겠다!" 싶을 수 있습니다. 근데 이건 내 컴퓨터에서만 열리는 주소입니다. 이걸 모르고 로컬 주소를 공유했다가 "나 안 보이는데?"라는 답장을 받는 건... 다들 한번쯤 겪는 통과의례입니다.

### 배포하면 따라오는 것들

**도메인**(Domain) — `myapp.com` 같은 주소입니다. 식당의 간판이자 주소.

**DNS**(Domain Name System) — "myapp.com"을 실제 서버 위치로 안내해주는 시스템입니다. 인터넷의 전화번호부라고 보면 됩니다.

**SSL/HTTPS** — 데이터를 암호화해서 안전하게 주고받는 기술입니다. 주소창의 자물쇠 아이콘이 바로 이것. 이게 없으면 누군가 중간에서 비밀번호나 결제 정보를 엿볼 수 있습니다.

이것들을 직접 설정하는 건 복잡하지만, Vercel이나 Netlify 같은 배포 서비스를 쓰면 대부분 자동으로 처리됩니다.

**이런 일이 생겼을 때 이 지식이 쓰입니다.** 배포하고 나서 "https가 안 붙어요", "도메인 연결했는데 접속이 안 돼요" 같은 문제가 생깁니다. 이때 코드를 뒤지기 시작하면 한도 끝도 없습니다. 이게 코드 문제가 아니라 **인프라 문제**라는 걸 아는 것만으로, 삽질 시간이 확 줄어듭니다.

---

## 빈 곳이 보이면 어디가 빠졌는지도 보인다

5가지를 전부 열어봤으니, 이제 각각이 빠지면 어떤 일이 벌어지는지 한눈에 정리해봅시다.

| 빠진 것 | 앱에서 벌어지는 일 |
|---|---|
| 프론트엔드 | 화면이 없으니 사용자가 아무것도 할 수 없음 |
| API | 프론트엔드가 백엔드에 요청을 보낼 방법이 없음 |
| 백엔드 | 데이터는 있는데 "누구 것인지" 판단하는 로직이 없음 |
| DB | 새로고침하면 모든 데이터가 사라짐 |
| 인프라 | 내 컴퓨터에서만 돌아감. 아무도 접속 못 함 |

---

## 도구 지도 — "지금 뭐가 채워져 있는 거지?"

5가지 구성요소를 배웠으니, 이제 본론입니다. 당신이 이미 쓰고 있거나 들어본 도구들이 이 지도의 어디에 꽂히는지 확인해봅시다.

| 도구 | 프론트엔드 | API | 백엔드 | DB | 배포 | 성격 |
|---|:---:|:---:|:---:|:---:|:---:|---|
| **Supabase** | — | ✅ | ✅ | ✅ | ✅ | DB + 인증 + API를 한번에 |
| **Firebase** | — | ✅ | ✅ | ✅ | ✅ | 구글 버전의 올인원 백엔드 |
| **Vercel** | ✅ | ✅ | ✅ | — | ✅ | 프론트 배포 + 서버리스 함수 |
| **Lovable / Bolt** | ● | ● | ● | ● | △ | 프롬프트로 풀스택 앱 코드 생성 |
| **v0** | ● | — | — | — | △ | 프론트엔드 UI 생성 특화 |
| **Cursor** | ○ | ○ | ○ | ○ | — | 개발자용 AI 코드 에디터 |

_✅ 직접 실행·운영 · ● AI가 코드 생성 · △ 제한적 · ○ 사용자가 직접 구현 · — 미지원_

여기서 중요한 구분이 있습니다. **Supabase, Vercel, Firebase는 코드가 "실행되는" 플랫폼**이고, **Lovable, v0, Cursor는 코드를 "만들어내는" 도구**입니다. AI 빌더가 코드를 만들어주면, 그 코드는 Supabase나 Vercel 같은 플랫폼 위에서 돌아갑니다.

### 그래서 현실에서는 이런 조합이 나옵니다

> **AI 빌더(코드 생성) + Next.js(프론트엔드) + Supabase(DB+백엔드) + Vercel(배포)** = AI 빌더 1개 + 플랫폼 3개로 5가지가 전부 채워짐

이것이 현재 바이브코딩 커뮤니티에서 가장 많이 검증된 조합입니다. AI가 가장 많이 학습한 도구 조합이라 코드 생성 품질이 높고, 무료 요금제로 MVP까지 충분히 만들 수 있습니다. **AI가 잘 아는 도구를 쓰는 것 자체가 전략입니다.**

---

## 도구가 새로 쏟아져 나와도 당황하지 않는 법

바이브코딩을 하다 보면, 낯선 이름을 끊임없이 마주칩니다. Clerk, Stripe, Resend, Upstash... 처음엔 하나하나 새로운 게 나올 때마다 "또 뭐야" 싶고 지칩니다. 근데 이 질문 하나를 습관으로 만들면 달라집니다:

> **"이 도구는 5가지 구성요소 중 어디를 채우는가?"**

Clerk → "인증(백엔드의 일부)을 전문으로 하는 서비스구나."
Stripe → "결제 처리(백엔드의 일부)를 담당하는 서비스구나."
Resend → "이메일 발송(백엔드의 일부)을 해주는 서비스구나."

이렇게 지도 위에 꽂을 수 있으면, 새 도구가 쏟아져 나와도 당황하지 않습니다. **도구가 바뀌어도 지도는 안 바뀝니다.**

---

## AI에게 더 잘 시키는 법: 수직 슬라이스

5가지 구성요소를 알게 되면, AI에게 지시하는 방식도 달라집니다.

AI에게 "할일 앱 전체를 만들어줘"라고 한 번에 시키면, 결과물이 뒤죽박죽이 될 확률이 높습니다. 대신, 기능 하나를 **프론트엔드부터 데이터베이스까지 수직으로 완성**하는 방식이 효과적입니다.

1. "할일을 추가하는 기능을 만들어줘. 입력 폼(프론트엔드), Supabase에 저장하는 API, todos 테이블까지."
2. 이게 완전히 돌아가는 걸 확인한 다음 →
3. "할일을 삭제하는 기능을 추가해줘."

이 방식을 **수직 슬라이스(Vertical Slice)**라고 합니다. 케이크를 한 층씩 쌓는 게 아니라, 얇은 조각으로 잘라서 한 조각 안에 모든 층이 다 들어있게 만드는 것입니다. 이렇게 하면 각 단계에서 AI의 결과물을 확인하고, 문제가 생기면 바로 잡을 수 있습니다.

1. 프론트엔드를 만드는 HTML, CSS, JavaScript가 각각 "뼈대, 옷, 근육" 중 어떤 역할인지 말할 수 있나요?
2. REST API의 4가지 구성요소(URL, 메서드, 헤더, 바디)를 할일 앱 예시로 설명할 수 있나요?
3. SQL과 NoSQL의 차이를 "엑셀 시트 vs 스크랩북"으로 설명하고, 내 서비스에는 어떤 게 맞을지 판단할 수 있나요?
4. Supabase를 연결하면 5가지 중 어디가 채워지는지 말할 수 있나요?
5. **지금 실행 액션:** 폰에 깔린 앱 아무거나 하나 열어서, 버튼을 하나 눌러보세요. 그 순간 5가지 구성요소가 각각 뭘 하고 있는지 떠올려보세요. (예: 인스타그램에서 "좋아요" 누르기 — 프론트엔드는 하트 애니메이션, API는 POST 요청, 백엔드는 "이 사용자가 이 게시물에 좋아요를 누른 것" 확인, DB는 좋아요 기록 저장, 인프라는 이 모든 게 전 세계에서 동시에 작동하게 유지) 이게 되면 졸업입니다.

## 다음으로

이제 각 구성요소가 뭔지, 어떤 도구가 어디를 채우는지 알게 되었습니다. 그런데 이 다섯 가지는 따로따로 존재하는 게 아닙니다. 사용자가 버튼 하나를 누르면, 데이터가 프론트엔드 → API → 백엔드 → 데이터베이스를 거쳐 다시 돌아오는 **여행**을 합니다. 이 여행 중간에 뭔가 잘못되면 에러가 터지는데 — 챕터 3에서는 이 여행의 전체 경로를 따라가면서, "에러가 터졌을 때 어디를 봐야 하는지" 추적하는 법을 배웁니다.

**더 궁금하다면:**
- Supabase는 오픈소스이고, Firebase는 구글의 독점 서비스입니다. 장기적으로 "다른 곳으로 옮길 수 있는 자유"가 중요하다면 Supabase가 유리합니다.
- 서비스가 커져서 Vercel의 서버리스 함수로 부족해지면 **Railway** 같은 범용 백엔드 인프라를 검토할 수 있습니다. 실제로 대역폭 비용 때문에 이전하는 사례도 있습니다.
- 서버리스 함수가 오랫동안 호출되지 않으면 처음 실행 시 잠깐 느려지는 **콜드 스타트(Cold Start)** 현상이 있습니다. "왜 처음 한 번만 느리지?" 싶으면 그게 콜드 스타트입니다.
