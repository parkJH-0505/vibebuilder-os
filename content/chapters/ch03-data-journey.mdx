## 버튼을 눌렀는데 아무 일도 안 일어난다

회원가입 기능을 넣었습니다. 이메일과 비밀번호를 입력하고 "가입" 버튼을 눌렀습니다. 화면에서는 아무 일도 안 일어납니다. 에러 메시지도 없고, 성공 메시지도 없고, 그냥... 아무 일도요.

"프론트엔드가 문제야? API가 문제야? 데이터베이스에 저장이 안 된 거야?"

이 질문에 답할 수 없으면 AI한테 "안 되는데 고쳐줘"만 반복하게 됩니다. AI는 이것저것 제안하지만 맥락이 없으니까 빗나가는 경우가 많습니다.

> "AI가 제시하는 해결책을 계속 적용했지만, 하나를 고치면 다른 것이 망가지는 악순환에 빠졌다." — r/vibecoding 사용자 경험담


5가지 구성요소를 '따로따로' 아는 것과, 그것들이 '함께 작동하는 흐름'을 아는 건 완전히 다른 이야기입니다. 이 챕터에서는 그 장기들 사이로 **데이터가 어떻게 흘러가는지** — 데이터의 여행 경로를 따라갑니다.

## 카카오톡 메시지의 여행

웹에서 데이터가 오고 가는 방식의 핵심은 택배를 보내고 받는 것과 같습니다.

- **요청(Request)** = 택배를 보내는 것. "이 데이터 줘" 또는 "이거 저장해줘"
- **응답(Response)** = 택배 회신을 받는 것. "여기 있어" 또는 "저장 완료했어" 또는 "그건 못 해"

**항상 브라우저가 먼저 요청하고, 서버가 응답합니다.** 서버가 먼저 보내지는 않습니다. 이제 구체적으로 따라가봅시다. 카카오톡에서 "안녕"이라고 치고 전송 버튼을 누르는 그 순간, 6단계의 여행이 시작됩니다.


**1단계: 사용자가 버튼을 누른다 (프론트엔드)**

"전송" 버튼을 탭합니다. 프론트엔드가 이 클릭을 감지합니다.

**2단계: 택배를 포장한다 (HTTP 요청 생성)**

프론트엔드가 택배 상자를 포장합니다. 챕터 2에서 배운 4가지 정보가 들어갑니다:
- **배송 라벨(URL)**: `/messages`라는 주소
- **배송 유형(HTTP 메서드)**: 새 메시지를 보내니까 `POST`
- **통행증(Headers)**: "나는 로그인한 회원이다"라는 인증 토큰
- **내용물(Body)**: 보내는 메시지 "안녕", 받는 사람 ID, 보낸 시간

**3단계: 택배를 보낸다 (요청 전송)**

포장된 택배가 인터넷을 통해 카카오톡 서버로 전달됩니다. 눈 한번 깜빡이는 사이.

**4단계: 물류센터가 처리한다 (백엔드)**

카카오톡 서버가 도착한 택배를 엽니다. "이 사람이 진짜 로그인한 회원인지" 확인하고, 메시지를 **데이터베이스에 저장**하고, 받는 사람에게 알림을 보냅니다.

**5단계: 회신 택배를 보낸다 (HTTP 응답)**

서버가 처리를 마치면, 결과를 **상태 코드**(숫자 세 자리)와 함께 돌려보냅니다.

**6단계: 화면에 반영된다 (프론트엔드)**

프론트엔드가 회신을 받고, 화면에 내가 보낸 메시지가 말풍선으로 나타납니다.

여기까지가 **하나의 사이클**입니다. 전송 버튼을 누른 순간부터 말풍선이 뜨기까지, 이 6단계가 눈 깜짝할 사이에 일어난 겁니다.

> 중요한 함정: **6단계에서 프론트엔드가 응답을 제대로 받았어도, 화면을 바꾸는 코드가 없으면 화면은 그대로입니다.** 데이터의 여행은 성공했는데, 여행이 끝난 후의 이야기를 아무도 안 해놓은 상황.

## 단서: 서버가 남긴 숫자 세 자리

서버가 돌려보내는 회신에는 항상 숫자 세 자리가 붙어 있습니다. 이게 **상태 코드**(Status Code)입니다.

| 상태 코드 | 서버의 답변 | 식당 비유 |
|---|---|---|
| **200** OK | "요청 잘 처리했습니다" | "주문하신 음식 나왔습니다" |
| **401** Unauthorized | "신분 확인이 안 됩니다" | "예약 확인이 안 되는 손님은 입장 불가입니다" |
| **403** Forbidden | "그건 안 됩니다" | "이 메뉴는 VIP 회원만 주문 가능합니다" |
| **404** Not Found | "그런 건 없는데요" | "메뉴판에 없는 요리를 주문하셨네요" |
| **500** Internal Server Error | "저희 쪽에 문제가 생겼습니다" | "주방 오븐이 갑자기 고장 났습니다" |


규칙은 단순합니다:
- **2xx** = 성공.
- **4xx** = 내(요청하는 쪽) 잘못. 주소 오류, 권한 없음, 형식 틀림.
- **5xx** = 서버 잘못. 서버 쪽에서 뭔가 터진 것.

**401과 403의 차이가 중요합니다.** 401은 "신원 미확인" — 로그인 안 했거나 API 키가 없는 경우. 403은 "신원 확인됐지만 권한 없음" — 로그인은 했지만 자격이 없는 경우.


이 숫자 하나가 "어디를 봐야 하는지"를 바로 알려줍니다.

## 현장: 배달의민족에서 짜장면 주문하기

같은 추적 방식을 다른 서비스에 적용해봅시다.

**1단계 — 프론트엔드:** 앱에서 짜장면을 장바구니에 넣고 "결제하기"를 누릅니다.

**2단계 — API 요청:** "이 사용자가 이 식당에서 짜장면 1개를 카드로 결제하려 합니다"를 담아서 `POST /orders`로 보냅니다.

**3단계 — 백엔드 처리:** 서버가 회원 확인, 영업 중인지 확인, 그리고 결제 요청을 **외부 결제 서비스(토스페이먼츠 같은)에 전달**합니다. 여기서 중요한 게 있습니다 — **배달의민족 서버가 또 다른 서버에 요청을 보냅니다.** 데이터의 여행은 한 방향이 아닙니다. 서버가 다른 서버에 요청하고, 그 응답을 받아 다음 일을 처리하는 연쇄 구조입니다. 이 연쇄 중 어디 한 곳이 막히면 전체가 실패합니다.

**4단계 — DB 저장:** 결제 성공 시 주문 내역이 데이터베이스에 기록됩니다. 이 기록이 있어야 식당 앱에 주문이 뜨고, 라이더 앱에 배달 요청이 뜹니다.

**5단계 — 응답 + 화면 업데이트:** 서버가 200을 돌려보내고, 내 폰 화면이 "주문이 접수되었습니다"로 바뀝니다.

여기서 핵심은 **서버가 또 다른 서버에 요청을 보내는** 멀티-홉 구조입니다. 배달의민족 서버가 결제 서비스, 식당, 라이더를 모두 연결하는 흐름을 보세요:

### 인스타그램 "좋아요"는 조금 다릅니다

하트를 누르면 **서버 응답을 기다리지 않고 화면을 먼저 바꿔버립니다.** "낙관적 업데이트"라고 합니다. 1~2초 동안 반응이 없으면 사용자가 또 누르니까, "일단 바꿔놓고, 실패하면 되돌린다"는 전략입니다. 이런 판단도 데이터 흐름을 이해해야 내릴 수 있습니다.

## 4가지 함정: 바이브코딩에서 가장 많이 빠지는 곳

데이터의 여행 경로 어딘가에서 막혔을 때, 주로 나타나는 패턴 네 가지입니다.

### 함정 1: 조용한 실패 (Silent Failure)

가장 무서운 유형입니다. 에러 메시지가 없습니다. 아무 일도 안 일어납니다. 가입 버튼을 눌렀는데 아무 반응이 없다? 십중팔구 **요청 자체가 안 나가고 있거나, 응답을 받고 나서 아무것도 안 하거나** 둘 중 하나입니다.

더 무서운 케이스도 있습니다. AI를 이용해 시스템을 만들었는데 코드가 **랜덤한 가짜 데이터를 생성하고 있었을 뿐**인 사례. 에러가 없으니 작동한다고 착각한 겁니다.


**진단:** 브라우저에서 F12 → Network 탭을 열고 버튼을 누릅니다. 요청이 안 뜨면 프론트엔드 문제, 뜨면 상태 코드를 봅니다.

### 함정 2: CORS라는 벽

내 컴퓨터에서 개발할 때는 잘 됩니다. 배포하는 순간 API 요청이 전부 막힙니다.

**내 도메인이 아닌 다른 도메인으로 요청을 보낼 때, 서버가 명시적으로 허락해줘야 한다**는 브라우저 보안 규칙 때문입니다. 로컬(`localhost:3000`)과 배포된 주소(`myapp.com`)는 브라우저 입장에서 다른 "출처"이기 때문에 이런 일이 생깁니다.


**핵심:** CORS는 **서버 측 설정** 문제입니다. 프론트엔드 코드를 아무리 고쳐도 해결되지 않습니다.

### 함정 3: 환경 변수의 배신 (Ch.5에서 자세히 다룹니다)

로컬에서는 API 연결이 완벽하게 됩니다. 배포하는 순간 401 에러가 납니다.

이유: 로컬에서는 `.env.local` 파일에 API 키를 저장합니다. 이 파일은 배포 서버로 자동으로 가지 않습니다. Vercel 같은 플랫폼의 대시보드에서 **따로 환경 변수를 설정하고, 재배포를 해야** 적용됩니다. 재배포를 안 하면 입력해도 적용이 안 됩니다. 이 순서를 놓쳐서 한 시간을 날리는 사례가 정말 많습니다.

**진단:** 배포 후 갑자기 401이 뜨면, 배포 플랫폼 대시보드에서 환경 변수 설정과 재배포 여부를 확인합니다.

### 함정 4: 로그인됐는데 화면이 안 바뀐다

데이터 흐름의 모든 단계가 성공했는데, 마지막 단계 — **화면을 업데이트하는 코드** — 가 빠진 경우입니다. 로그인 성공 응답을 받았는데 헤더 버튼이 안 바뀌고, 보여야 할 메뉴가 안 보이고. 서버/백엔드 문제가 아닙니다.

**진단:** AI에게 "로그인 API는 200 OK가 뜨는데, 로그인 후 헤더 UI가 바뀌지 않습니다. 로그인 성공 후 사용자 세션 상태를 업데이트하고 헤더를 다시 렌더링하는 코드를 추가해주세요."

## 막혔을 때 어디를 봐야 하는가

에러가 뜨면 많은 사람이 그것을 그대로 AI에 복사-붙여넣기 합니다. 하지만 에러 메시지는 **문제를 해결할 단서**입니다. **What**(무엇이 잘못됐나), **Where**(어디서), **Why**(왜) — 이 세 가지를 읽고 AI에게 전달하면 완전히 다른 답변이 나옵니다.


가장 강력한 도구는 브라우저에 이미 내장되어 있습니다. F12를 누르고 **Network 탭**을 열면, 어떤 요청이 어디로 나갔고 서버가 뭐라고 답했는지 전부 볼 수 있습니다.


에러를 만났을 때 이 순서대로 확인하세요:

**1. 터미널 먼저 본다** — 개발 서버 출력에 원인이 직접적으로 나오는 경우가 많습니다.

**2. Network 탭에서 요청이 나갔는지 본다** — 안 나갔으면 프론트엔드 문제.

**3. 상태 코드를 본다** — 200이면 성공, 4xx면 내 쪽, 5xx면 서버 쪽.

**4. 응답(Response) 데이터를 본다** — 데이터가 있는데 화면에 안 보이면 프론트엔드 렌더링 문제.

### AI에게 에러를 전달하는 올바른 방법

**이렇게 하면 안 됩니다:**
"데이터가 안 보여요. 고쳐주세요."

**이렇게 해야 합니다:**
"회원가입 버튼을 누르면 네트워크 탭에 `POST /api/auth/signup` 요청이 나가는데, 상태 코드가 **401**입니다. Headers를 보면 Authorization 항목이 비어있습니다. 환경 변수에 `SUPABASE_ANON_KEY`가 설정되어 있는지 확인해주세요."

이 정보량의 차이가 AI가 줄 수 있는 답의 질을 결정합니다.

## 다음으로

데이터가 흘러가는 경로를 이제 알게 되었습니다. 그런데 이 여행에는 "신분 확인"이라는 관문이 있습니다. 카카오톡 메시지 4단계에서 "이 사람이 진짜 로그인한 회원인지 확인한다"고 했는데 — 이게 어떻게 작동하는지는 아직 안 다뤘습니다. 챕터 4에서는 인증과 권한의 세계로 들어갑니다. 로그인이 기술적으로 어떻게 작동하는지, 그리고 왜 이걸 직접 만들면 위험한지를 알게 됩니다.

**더 궁금하다면:**
- HTTP 상태 코드는 실제로 수십 가지가 있지만, 실전에서 마주치는 건 200, 301, 400, 401, 403, 404, 500 정도입니다. 나머지는 만나면 그때 검색하면 됩니다.
- 환경 변수 배신 패턴은 너무 흔해서, Vercel 배포 후 작동 안 하는 문제의 절반 이상이 이것입니다.
- 복잡한 앱을 만들 때는 AI에게 "개발 환경에서만 API 응답 상태와 인증 상태를 실시간으로 표시하는 작은 패널을 추가해줘"라고 부탁할 수 있습니다. 네트워크 탭을 매번 열지 않아도 데이터 흐름을 확인할 수 있습니다.
