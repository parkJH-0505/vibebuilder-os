## 도구가 너무 많아서 아무것도 못 시작했습니다

커뮤니티에서 "바이브코딩 어떤 도구로 시작해요?"라고 물으면, 10개의 댓글이 달립니다. Lovable 추천, Cursor 추천, Bolt 써봤는데 좋더라, Claude Code가 최고다, Windsurf도 있다, v0는 어떻게 생각해요?

읽을수록 혼란스럽습니다.

결국 많은 분들이 이렇게 됩니다. 세 개를 설치하고, 각각 뭔가 해보다가, 어느 게 맞는지 모르겠고, 괜히 옮겨타기 반복하다가, "나중에 제대로 알아봐야지" 하고 탭을 닫습니다.

이게 **분석마비(Analysis Paralysis)**입니다. 선택지가 많을수록 아무것도 선택 못 하는 현상. 바이브코딩 도구 선택은 이 함정이 특히 심합니다.

이 함정에서 나오는 방법은 단순합니다. **도구가 뭘 담당하는지 알면 비교할 수 있습니다. 비교할 수 있으면 고를 수 있습니다.**

그리고 이 챕터를 다 읽으면, 도구 선택 기준이 한 문장으로 정리됩니다.

---

## 먼저 기억할 것: 어떤 도구도 혼자서 다 해주지 않습니다

Part 1에서 배웠던 프로덕트의 5가지 구성요소를 떠올려보세요.

- **프론트엔드** — 사용자가 보고 만지는 화면
- **API** — 프론트엔드와 백엔드를 연결하는 통로
- **백엔드** — 화면 뒤에서 일어나는 처리 로직
- **데이터베이스** — 정보를 저장하는 곳
- **인프라/배포** — 이 모든 것을 인터넷에 올리는 것

진짜 서비스가 되려면 이 다섯 가지가 다 있어야 합니다.

그런데 바이브코딩 도구를 쓰면서 가장 많이 하는 착각이 있습니다. "이 도구 하나면 다 됩니다"라는 마케팅 문구를 믿고 시작했다가, 어느 순간 막히는 것. 데이터가 저장이 안 된다거나, 배포하면 작동이 안 된다거나.

이건 도구가 나쁜 게 아닙니다. 그 도구가 담당하는 구성요소와 담당 안 하는 구성요소가 있는데, 그 선을 몰랐던 거입니다.

실제로 v0를 쓴 한 사용자 경험담이 있습니다. 폼에 데이터를 입력하고 '저장' 버튼을 눌렀는데, 새로고침하면 데이터가 사라집니다. v0가 프론트엔드 화면은 완벽하게 만들어줬지만, 데이터를 어디에 저장할지는 연결이 안 된 상태였던 겁니다. "v0는 아름다운 레이아웃을 생성하지만, 실제 로직을 연결하고, 외부 API를 연결하고, 다른 곳에 백엔드를 배포해야 합니다."

이 챕터는 각 도구가 5가지 구성요소 중 어디를 담당하는지를 정확히 보여줍니다.

---

## 도구 선택 기준은 한 문장입니다

먼저 이 기준을 잡고 들어갑니다.

> **"내가 만들고 싶은 기능이 어느 장기까지 필요한지가 곧 도구 선택 기준이다."**

프론트엔드(화면)만 필요하면 → 브라우저 기반 도구로 충분합니다.
백엔드와 데이터베이스까지 필요하면 → IDE/CLI 기반 도구로 가야 합니다.
코드는 완성됐고 배포만 남았으면 → 배포 플랫폼이 필요합니다.

이 기준 하나만 있으면, 도구 선택이 추측이 아니라 판단이 됩니다.

이제 이 기준을 바탕으로 실제 도구를 살펴보겠습니다.

---

## 도구는 세 종류입니다 — 전체 지도

바이브코딩 도구를 크게 셋으로 나눌 수 있습니다.

**브라우저 기반 도구** — 브라우저에서 열고, 프롬프트를 치면 화면을 만들어주는 도구. Lovable, Bolt.new, v0가 여기 해당합니다. 진입 장벽이 가장 낮습니다. 설치 없이 바로 시작할 수 있습니다.

**IDE 기반 도구** — 코드 편집기(IDE) 안에서 AI와 함께 작업하는 도구. Cursor, Windsurf가 대표적입니다. 자신의 프로젝트 전체를 AI가 이해하며 도와줍니다.

**CLI 기반 도구** — 터미널(명령어 입력창)에서 자연어로 지시하는 도구. Claude Code가 대표적입니다. 가장 강력하지만, 터미널에 익숙하지 않으면 진입이 어렵습니다.

큰 그림을 먼저 보면 이렇습니다.

| | 프론트엔드 | 백엔드 | 데이터베이스 | 배포 | 진입 난이도 |
|---|---|---|---|---|---|
| **Lovable** | ✅ | ❌ | ❌ (외부 연결) | ❌ | 쉬움 |
| **Bolt.new** | ✅ | △ | △ | ❌ | 쉬움 |
| **v0** | ✅ | ❌ | ❌ | ❌ | 쉬움 |
| **Cursor** | ✅ | ✅ | ✅ | ❌ | 보통 |
| **Windsurf** | ✅ | ✅ | ✅ | ❌ | 보통 |
| **Claude Code** | ✅ | ✅ | ✅ | ❌ | 어려움 |
| **Vercel/Netlify** | ❌ | ❌ | ❌ | ✅ | 쉬움 |
| **Railway** | ❌ | △ | ✅ | ✅ | 보통 |
| **Supabase** | ❌ | △ | ✅ | ❌ | 보통 |

여기서 바로 보이는 것이 있습니다. **배포를 담당하는 도구는 따로 있습니다.** 코드를 만드는 도구와 배포하는 도구는 다릅니다. 대부분의 경우 둘 이상을 조합해야 완전한 서비스가 됩니다.

> **지금 바로 해보기:** 표를 보면서, 내가 만들 앱에 필요한 구성요소에 ✅를 찍어보세요. 어느 행이 가장 많이 겹치는 도구가 당신의 1순위 후보입니다.

이제 비교 기준이 생겼습니다. 각 카테고리를 구체적으로 살펴보겠습니다.

## 브라우저 기반 도구: 빠른 시작, 명확한 한계

### Lovable — 화면은 10분, 그 이후는 당신 몫

Lovable은 프롬프트 하나로 수 분 내에 완성도 높은 화면을 만들어줍니다. React 기반 코드를 자동으로 생성하고, 결과물을 바로 미리보기로 확인할 수 있습니다.

처음 써보면 솔직히 놀랍니다. "이게 이렇게 쉬웠어?" 싶은 순간이 분명히 옵니다. 그러다 어느 순간 막힙니다. 데이터를 저장하려는 순간.

**담당하는 것:**
- 프론트엔드 ✅ — 강력합니다. 버튼, 폼, 목록 화면을 빠르게 만들어줍니다.
- API 기본 구조 △ — 아주 기본적인 골격은 만들어줍니다. 복잡한 비즈니스 로직은 어렵습니다.

**담당 안 하는 것:**
- 백엔드 로직 ❌ — 복잡한 처리, 실시간 기능은 직접 구현해야 합니다.
- 데이터베이스 ❌ — Supabase라는 외부 서비스를 연결해야 합니다. 구글 소셜 로그인을 붙이려면 GCP(Google Cloud Platform)에서 프로젝트를 생성하고, 인증 키를 발급받아 Supabase에 입력하는 과정이 필요합니다.
- 배포 ❌ — 별도 설정이 필요합니다.

**이럴 때 쓰면 딱입니다:** 아이디어를 빠르게 화면으로 보여주고 싶을 때. "이렇게 생긴 서비스입니다"를 투자자나 팀에게 보여주는 프로토타입용으로 탁월합니다. 실제 데이터를 저장하는 서비스를 만들려면 Supabase 연결이 필요하다는 것을 알고 시작하면, 막히는 지점이 줄어듭니다.

### Bolt.new — 기본 풀스택, 한 단계 더

Bolt.new는 Lovable보다 한 단계 더 나아갑니다. 자체 데이터베이스 기능이 포함되어 있고, 기본적인 백엔드 로직도 생성해줍니다.

**담당하는 것:**
- 프론트엔드 ✅ — Lovable과 비슷한 수준
- 백엔드 기본 로직 △ — 간단한 처리는 됩니다
- 데이터베이스 △ — 자체 DB가 있지만, 안정적인 서비스를 위해서는 Supabase 연동을 권장합니다

**담당 안 하는 것:**
- 배포 ❌ — 별도 설정 필요

**이럴 때 쓰면 딱입니다:** 프로토타입보다 한 단계 나아가 실제로 작동하는 간단한 서비스를 만들고 싶을 때. 백엔드까지 포함해서 빠르게 돌아가는 걸 만들고 싶다면 Lovable보다 한 단계 위 선택입니다.

### v0 (by Vercel) — UI 컴포넌트 전문가

v0는 솔직합니다. "풀스택 생성기가 아닌 프론트엔드 UI 생성기"라는 정체성이 명확합니다.

**담당하는 것:**
- 프론트엔드 ✅ — shadcn/ui와 Tailwind CSS 기반의 고품질 UI 컴포넌트. 개발자들이 "v0는 정말 좋다"고 하는 이유입니다.

**담당 안 하는 것:**
- 백엔드, 데이터베이스, 배포 ❌ — 코드 제안은 해줄 수 있지만, 실제 실행은 사용자가 직접 해야 합니다.

**이럴 때 쓰면 딱입니다:** 이미 프로젝트가 있고, 특정 UI 화면이나 컴포넌트를 빠르게 만들어 붙이고 싶을 때. 처음 시작하는 분이 "앱 전체를 만들 도구"로 접근하면 막힙니다. v0는 화면 전문가입니다.

## IDE 기반 도구: 강력하지만, 진입 장벽이 있습니다

### Cursor — 코드를 보면서 함께 작업

Cursor는 VS Code를 기반으로 만들어진 AI 코드 편집기입니다. 브라우저 기반 도구와 결정적으로 다른 점은, **당신의 프로젝트 전체를 AI가 읽고 이해하면서** 작업한다는 것입니다.

새 기능을 추가할 때, 기존 코드의 구조와 일관성을 유지하면서 추가합니다. 버그를 고칠 때, 그 코드가 다른 부분과 어떻게 연결되어 있는지 파악하면서 고칩니다. 브라우저 기반 도구에서는 이게 안 됩니다. 프로젝트 전체 맥락을 모르니까요.

**담당하는 것:**
- 코드 생성·수정·리팩토링 ✅ — 프론트엔드, 백엔드, 데이터베이스 연결 코드 전부
- 버그 수정, 기능 추가 ✅

**담당 안 하는 것:**
- 배포 ❌ — 코드를 만들어주지만, GitHub에 올리고 Vercel로 배포하는 과정은 직접 해야 합니다
- 인프라 설정 ❌

**이럴 때 쓰면 딱입니다:** 이미 프로젝트가 만들어진 상태에서, 기능을 추가하거나 수정하고 싶을 때. 코드 편집기 환경이 낯설다면 "처음에 Lovable로 화면을 만들고, 이후 기능 확장은 Cursor로"라는 조합을 많이 씁니다.

### Windsurf — Cursor와 비슷한 포지션

Cursor와 거의 같은 카테고리입니다. AI 네이티브 코드 편집기이고, 담당 범위도 비슷합니다. 2026년 현재 Cursor와 Windsurf는 "어느 쪽이 나은가"를 두고 커뮤니티에서 계속 비교되고 있습니다. 처음 IDE 기반 도구를 써본다면 둘 중 어느 쪽을 선택해도 크게 다르지 않습니다.

**이럴 때 쓰면 딱입니다:** Cursor와 동일합니다. 취향 차이라고 봐도 됩니다.

## CLI 기반 도구: 가장 강력하지만, 가장 가파릅니다

### Claude Code — 터미널에서 자연어로 지시

Claude Code는 터미널(명령어를 입력하는 검은 창)에서 작동합니다. Anthropic(Claude를 만드는 회사)이 만든 도구입니다.

솔직히 말하면, 터미널이 처음이라면 이 도구는 지금 당장 쓰기 어렵습니다. 그건 도구가 나쁜 게 아니라, 진입 시점의 문제입니다. 바이브코딩에 어느 정도 익숙해진 뒤에 자연스럽게 옮겨오게 되는 도구입니다.

**담당하는 것:**
- 전체 프로젝트 코드 생성·수정 ✅
- 프로젝트 구조 이해 및 일관성 유지 ✅ — `CLAUDE.md`라는 파일에 프로젝트 규칙을 적어두면, AI가 그 규칙을 따르면서 작업합니다. Ch.10에서 이 파일 활용법을 자세히 다룹니다.
- Supabase, Figma 등 외부 서비스 연동 지원 ✅

**담당 안 하는 것:**
- 배포 ❌ — 코드를 만들어준 뒤 배포는 별도

**이럴 때 쓰면 딱입니다:** 어느 정도 바이브코딩에 익숙해졌고, 더 복잡한 프로젝트를 만들고 싶을 때. 터미널이 처음이라면 브라우저 기반 도구로 먼저 시작하기를 권장합니다.

## 배포 플랫폼: 마지막 퍼즐 조각

브라우저 기반이든 IDE 기반이든 CLI 기반이든, 모든 바이브코딩 도구가 공통으로 담당 안 하는 것이 있습니다. **배포** — 코드를 실제 인터넷에 올려서 다른 사람이 쓸 수 있게 하는 것.

**Vercel** — Next.js 프로젝트에 가장 최적화되어 있습니다. GitHub에 코드를 올리면 자동으로 배포됩니다. 개인 프로젝트 무료 범위가 넉넉합니다. (단, 상업적 이용 시 유료 플랜 확인 필요)

**Netlify** — Vercel과 비슷합니다. 정적 사이트나 간단한 프론트엔드 배포에 강합니다.

**Railway** — 백엔드 서버가 필요한 프로젝트에 적합합니다. 데이터베이스까지 함께 올릴 수 있습니다.

"코드 만드는 도구 + 배포하는 도구"는 별개입니다. 이 둘을 같은 도구에서 해결하려고 하면 막힙니다. 배포 설정 방법은 Ch.8에서 이미 다뤘습니다.

## 도구 선택 프레임: 두 가지 기준

지금까지 살펴본 도구들을 고를 때, 기준이 두 가지입니다.

**기준 1: 목적 — 무엇을 만드는가**

| 목적 | 추천 도구 |
|---|---|
| 화면만 빠르게 보여주고 싶다 | Lovable, v0 |
| 실제로 데이터가 저장되는 서비스 | Lovable + Supabase, Bolt.new |
| 기존 프로젝트에 기능 추가 | Cursor, Windsurf |
| 복잡한 프로젝트, 세밀한 제어 | Cursor, Claude Code |
| 코드 완성, 배포만 남음 | Vercel, Netlify, Railway |

**기준 2: 숙련도 — 지금 내 수준은 어느 정도인가**

- 코딩을 한 번도 해본 적 없다 → 브라우저 기반 도구로 시작
- Lovable 등으로 화면을 만들어본 적 있다 → Bolt.new 또는 Cursor로 이동
- 코드를 보고 수정할 수 있다 → Cursor, Windsurf
- 터미널에 익숙하다 → Claude Code

이 두 기준을 교차하면, 선택지가 2~3개로 좁혀집니다. 그 중에서 취향껏 골라도 됩니다.

한 가지 더. 바이브코딩 도구 생태계는 빠르게 변합니다. 지금 이 표가 3개월 후에도 동일하리라는 보장이 없습니다. **도구를 고른 뒤에도, 3~6개월 단위로 새 도구가 나왔는지 확인하는 습관이 중요합니다.** 도구는 바뀝니다. 판단 기준은 바뀌지 않습니다.

## "그래서 나는 뭘 써야 해요?" — 상황별 가이드

### 처음 시작하는 분이라면

**Lovable + Supabase + Vercel** 조합을 권장합니다.

Lovable로 화면을 만들고, Supabase로 데이터베이스와 로그인을 연결하고, Vercel로 배포합니다. 이 셋이 2026년 현재 비개발자에게 가장 많이 추천되는 조합입니다. 각각 무료로 시작할 수 있고, 연동 가이드도 풍부합니다.

단, Supabase 연동 과정이 처음엔 복잡하게 느껴질 수 있습니다. 이 부분을 건너뛰면 나중에 보안 문제가 생깁니다. "백엔드 지식이 없는 바이브 코더라면 Supabase는 시한폭탄 그 자체"라는 말이 있습니다. 과장이 아닙니다. Ch.4에서 배운 인증과 Ch.5에서 배운 API 키 보안이 여기서 다시 중요해집니다.

### 프로토타입보다 완성도 있는 서비스를 원한다면

**Bolt.new + Supabase + Vercel** 또는 **Cursor + Supabase + Vercel**

Bolt.new는 백엔드 기본 로직까지 생성해줘서 한 단계 더 완성된 서비스를 만들 수 있습니다. Cursor는 코드를 직접 보고 수정할 수 있어서, 세밀한 제어가 필요할 때 적합합니다.

둘 중 어느 쪽을 선택할지 모르겠다면: "코드를 직접 볼 준비가 됐는가?" — "예"라면 Cursor, "아직은 아니다"라면 Bolt.new입니다.

### 이미 프로젝트가 있고 기능을 추가하고 싶다면

**Cursor 또는 Claude Code**

기존 코드베이스를 이해하면서 기능을 추가하는 것은 브라우저 기반 도구가 잘 못하는 영역입니다. IDE/CLI 기반 도구의 강점입니다. 기존 프로젝트 맥락을 AI에게 알려주고, 거기서 이어 작업하는 방식입니다.

## 도구보다 중요한 것 — 이게 이 챕터의 핵심입니다

솔직히 말하겠습니다. 여기까지 읽으면서 "그래서 나는 Lovable 쓰면 되겠구나"를 확인하러 온 분들이 있을 겁니다. 맞습니다, 그것도 중요합니다.

근데 더 중요한 게 있습니다.

**도구는 방법을 대체하지 않습니다.** 어떤 도구를 써도, 뭘 만들지 모르면 결과가 엉뚱하게 나옵니다. 어떻게 지시할지 모르면 도구가 힘을 못 씁니다.

커뮤니티를 보면 패턴이 있습니다. "Lovable 써봤는데 안 된다 → Bolt로 바꿈 → 거기서도 막힘 → Cursor 써봄 → 어렵다 → 포기." 이게 도구의 문제인 경우는 드뭅니다.

대부분 원인은 세 가지 중 하나입니다.

**첫째, 무엇을 만들려는지가 불명확합니다.** Ch.6에서 배운 PRD 없이 "앱 만들어줘"로 시작하면, 어떤 도구를 써도 결과가 엉뚱하게 나옵니다. AI는 모호한 지시를 나름대로 해석해서 만들어주는데, 그게 당신이 원하는 것과 다른 겁니다.

**둘째, AI에게 지시하는 방법이 비효율적입니다.** 같은 도구를 써도 결과물의 수준이 극적으로 달라집니다. 기획 세션과 코딩 세션을 어떻게 분리하는지, 에러가 났을 때 어떻게 대응하는지 — 이것이 Ch.10의 주제입니다.

**셋째, 한 번에 너무 많은 걸 시키고 있습니다.** "로그인, 상품 등록, 결제, 알림까지 다 만들어줘"는 인간 개발자에게 줘도 버거운 요구입니다. Ch.7 MVP 원칙대로, 핵심 기능 하나부터 시작합니다.

도구를 또 바꾸기 전에, 이 세 가지를 먼저 확인해보세요.

## 다음으로

도구를 골랐습니다. 조합도 알겠습니다.

이제 그 도구를 "제대로" 써야 합니다.

같은 Cursor를 써도, 어떻게 지시하느냐에 따라 결과물이 완전히 달라집니다. 막막할 때 "에러 고쳐줘"라고만 하면 AI도 막막합니다. 프로젝트 전체 맥락을 어떻게 AI에게 전달하는지, 에러가 났을 때 어떤 순서로 대응하는지, 그리고 같은 실수를 반복하지 않으려면 어떻게 기록해야 하는지 — 이 모든 것이 Ch.10의 주제입니다.
